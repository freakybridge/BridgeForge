## 一、本次改动意图（验收基准）

把两个重型 skill `debate`、`collab` 从"确认门开在烧完 token 之后"改为"渐进对齐、三道闸开在烧 token 之前"的六步流程，避免开工跑偏白烧 token。

经与用户两轮 `AskUserQuestion` 确认的设计决策：
1. **触发策略**：无条件每次都问（不智能跳过）——用户主动调这俩重型 skill 即意味着大任务。
2. 纳入四项优化点 **A / E / G / D**（见下）。

### 六步流程（两者统一）
```
闸① 引导发问 → 闸② 复述确认 → 研读(可追问) → [闸③ 方案/拆分大纲] → 执行 → 收尾
```
- **debate** 六步：引导发问 → 复述确认 → 研读(可追问) → 多轮辩论 → 方案大纲(闸③) → 实施
- **collab** 六步：引导发问 → 复述确认 → 研读(可追问) → 拆分大纲(闸③) → 并行执行 → 整合+独立验证

### 四项优化点
| 代号 | 归属 | 内容 |
|------|------|------|
| **A** | debate | 轮次上下文机制：后续轮用 **SendMessage 续接同一 A/B agent**（非每轮新起），MD 退回纯归档 |
| **E** | collab | 失败重试上限：子任务失败最多重试 1 次（共 2 次）即停下上报，对齐全局「2 次失败必停」红线 |
| **G** | 两者 | 轻量出口：对齐/复述阶段发现任务太小或不可并行拆，提前退出建议直接做 |
| **D** | collab | 整合后起**独立 review agent** 复查，不靠主 agent 自审 |

外加：collab 拆分粒度由闸① 偏好驱动（少而大/多而细），3-5 文件为上限；debate 共识后大改动建议转 `/collab`。

## 三、审计方法建议（针对本会话抽风教训）

1. **逐字读全文**：对文件 #1~#4，用 Read 工具读完整内容，人工核对六步结构与四项优化点文字是否真实存在、有无重影/乱码/占位垃圾（`待补充`/`占位文件`/重复标题）。

4. **git diff 复核**：`git status` + `git diff` 看本次工作区改动是否就是上述 6 个文件、内容是否符合预期（git 不经过抽风通道，是可靠的真相源之一）。
5. **判据红线**：任何"读出来有重影/数字自相矛盾/与预期矛盾"的结果，按 `CLAUDE.md §2.5` 一律二次验真，不在脏数据上下审计结论。
