# 讨论：跨文件重复的红线该放常驻层还是触发层

> 日期：2026-06-25
> 相关文件：`templates/CLAUDE.md`（§8 鬼打墙 / §11 doc 分层）、`templates/rules/debugging.md`（§6 鬼打墙）、`templates/rules/workflow.md`（§6 / §5.5 doc 分层）、`templates/rules/meta_rule_design.md`（§4.3 禁止同一约束多处重复 / §5 量化红线）
> 问题描述：产品层瘦身第二类。三个独立诊断 agent 都发现同一条红线在多个文件有**重复正文**——「鬼打墙升级」在 CLAUDE.md §8（每轮常驻）和 debugging.md §6（编辑 src/tests 才触发）两份；「doc 六层分层」在 CLAUDE.md §11、workflow.md §6 两份。meta_rule §4.3 自己禁止"同一约束多处重复"。该怎么去重？

## 共享事实（debate 双方共同前提）

1. **注入模式差异**：CLAUDE.md 每轮常驻 context；rule 按 frontmatter `paths:` 触发才加载（debugging 触发 `tests/** src/**`；workflow 触发 `doc/** .claude/rules/**`）。
2. **核心张力**：红线正文若从 CLAUDE.md 移到 rule → 省常驻 context，但在「不触发该 rule 的轮次」红线不在场（护栏覆盖漏洞）；若留 CLAUDE.md 删 rule 的 → 常驻成本不变但 rule 内少就近指引；若两边都留 → 违反 §4.3 且双份正文漂移风险。
3. **既有事实**：诊断发现 CLAUDE.md §11 已写"详见 workflow §6"——暗示 workflow 那份才是被指向的"正文"。
4. **鬼打墙的特性**：它是「任何任务反复失败」都该触发的元行为，不限于编辑 src/tests（改文档/配置反复失败也算）。

## Round 1

### 🔵 提议者
**两组方向相反，按「红线作用面」选层。**
- **鬼打墙**：正文留常驻 `CLAUDE.md §8`（鬼打墙是全场景元行为，任何文件反复失败都该触发，只有常驻层能覆盖）。`debugging.md §6` 删与 §8 重复的计数内核，只留 src/tests 特化增量（T3-T7 / native 诊断 / debate 启动 SOP），顶部加 pointer 指回 §8。paths 不扩面。
- **doc 分层**：正文收敛到触发 `workflow.md §5.5/§6`（红线作用面=建/改文档时，paths `doc/** .claude/rules/**` 精准覆盖）。`CLAUDE.md §11` 掏空为「一句最小红线 + pointer」，每轮省 ~28 行常驻（CLAUDE.md 已超 §5 的 ≤200 红线）。
- **顺带收益**：发现 §8 与 debugging §6 **计数阈值冲突**（§8「第 4 次硬停」vs debugging T1「≥2 次升级」）——去重消灭这个漂移活样本。

### 🔴 审阅者
**四个质疑 + 一个替代方案。**
1. **pointer 对红线是降级**：§9.x 外置的是「机制/配置」低频细节，红线正文一直就地。本次要搬的是红线**本身**，性质不同。meta_rule §1 强制力梯度（Rules > CLAUDE.md），pointer 比 rule 更弱一档——把红线变 pointer = 主动往梯度下方挪。
2. **护栏覆盖漏洞（关键）**：鬼打墙收敛 debugging（paths 仅 src/tests）→ 改 rule/hook/doc/纯对话推理时鬼打墙红线**整条缺席**（这些恰是高频试错区）。反之 doc 分层掏空 §11 → agent 在 `src/foo/` 里随手写 README 时，CLAUDE.md（被抽走）+ workflow（没触发 `doc/**`）**双双缺席**，正好在最该拦的场景失守。
3. **权威方向反了**：§11↔workflow §5.5 是**双向互指**（诊断只截一半）。且 meta_rule §4.4 明文「CLAUDE.md 放索引 + **必要红线**」——doc 六层是红线，本就该常驻；workflow 是从属的细节展开位。
4. **§4.3 真正含义**：§4.3 禁的是「**逐字复制同一正文**」（理由是"改一处忘另一处漂移"），不是「分工式共存」。meta_rule §9 自己背书「workflow vs meta_rule 互补不重叠」=同主题分工不算重复。
- **替代方案——红线分层共存**：常驻层（§8/§11）留**可执行红线骨架**（每轮在场），触发层（debugging/workflow）留**场景化操作细节**，只删两者间**逐字重叠的概述句**，不删不重叠的分工内容。补住覆盖空洞 + 不违 §4.3 + 与 §9.x 范式同构。

## Round 2

### 🔵 提议者（认账 + 落地清单）
- **撤回 R1「掏空 §11」**：承认覆盖漏洞成立——§11 掏空后，agent 在 `src/foo/` 写 README 时 CLAUDE.md（被抽走）+ workflow（没触发 `doc/**`）双双缺席，正好在唯一该拦的场景失守。
- **接受「红线分层共存」框架**，并落到行号级切分：
  - **doc 分层**：§11 **不掏空**，从 ~33 → ~12 行。留：标题红线 / 六层目录树（压成紧凑列表）/ 4 条禁止 / README 同步强制 / 立场句 / 末行 pointer。删：「为什么是红线」3 段论述（压成一行 Why）。workflow §5.5 强制项表（与 §11 4 禁止重叠）降为「操作细节 + Why 正文」。
  - **鬼打墙**：§8 留常驻（双方一致）。debugging §6 **不删计数内核**（R1 误判——§6 是 T1-T7 诊断矩阵 + SOP，非 §8 副本），只删开头「立即停手」近义句、顶部加 pointer 指回 §8。
- **阈值冲突修法**：主张「分层」——T1 改为 `2 次=预警(强制取量化数据)/ 4 次=硬停(对齐 §8)`，§8 不动。

### 🔴 审阅者（给判据 + 钉风险 + 裁决阈值）
- **骨架 vs 细节的机械判据**（reviewer 可套用）：**主判据**=「这条约束在『不触发该 rule 的轮次』也可能被违反吗？是→骨架常驻 / 否→细节下沉」；**副判据**=骨架只留「触发器 + 红线断言 + 唯一阈值」，操作流程 / 分类展开 / edge case / 论述一律下沉。
- **绝不下沉清单**（宁可超 200 行也常驻）：鬼打墙硬停断言+阈值、反「我有新思路」豁免句、六层目录树、4 条 doc 禁止、doc 唯一索引强制。§5 的 200 行是**软红线**，不能用它挤红线断言。
- **§5.5 反向 pointer 改向**：Why 论述正文权威落 workflow §5.5（它是 `doc/**` 触发层，论述放这天经地义），§11 只留断言 + 指针指向它。
- **阈值裁决**：主张统一取更严的 **2**（早停比晚停安全；专业版 debugging 在场时它说 2 就是 2；单值红线不能两处各写一个数）。反对提议者「2 预警/4 硬停」双层（怕下游 agent 困惑）。

## ✅ 共识结论
- **根因**：三个诊断 agent 报的「跨文件红线重复」，实为**骨架（粗粒度红线）+ 细节（场景化操作）的分工**，不是逐字复制。meta_rule §4.3 禁的是「同一正文复制两份→漂移」，**不禁**分工式共存（§9 自己背书「互补不重叠」）。所以正确动作不是「删一份/收敛单层」，而是**按覆盖面把红线骨架钉常驻、场景细节沉触发层，只删两者间逐字重叠句**。
- **切分判据**：主判据「非触发轮可否违反」一票定性（骨架/细节）；副判据切「骨架里留多少」（只留触发器+断言+阈值）。
- **方案（两组）**：
  1. **doc 分层**：`CLAUDE.md §11` 留红线骨架（六层树+4禁止+索引强制，~12 行），删 3 段论述压成一行 Why；`workflow.md §5.5` 降为操作细节 + 承载 Why 正文，删与 §11 重叠的强制项表正文。
  2. **鬼打墙**：`CLAUDE.md §8` 留常驻骨架（升级表+硬停+反豁免句）；`debugging.md §6` 留 T1-T7+SOP+5禁止，删开头「立即停手」近义句、加 pointer 指回 §8。
- **实施要点**：① 红线断言一律不降为纯 pointer（认 R1 质疑1/2/3）；省行数只能从论述/操作/分类表里省。② 六层树绝不下沉。③ CLAUDE.md 目标 ≤200 但以「不误伤骨架」为前提，超一点也可接受。
- **唯一待用户裁决分歧**：鬼打墙计数阈值统一成几（2 / 3 / 分层）——本质是用户容忍度（早停 vs 晚停成本），交用户拍板。
