# 防 AI 幻觉资源 · 四层纵深框架（方法论 + 出厂态决策）

> **定位**：这是「防 AI 幻觉资源」打法的**方法论 + 设计库**。红线条文（R1–R5）已落地为常驻 rule [`templates/rules/anti_fabrication.md`](../templates/rules/anti_fabrication.md)；本文存**不该进 rule 的部分**——四层框架、切分判据、三个 hook 的设计与「为什么不进产品层」的决策。
> **来源**：CausisRiskSuite 一次「幻觉文件名 → 工具空转 → 纯文字嫁祸」真实事故，经两轮跨项目对抗式 debate 定稿。交接全文见同目录 `handoff_2026-06-30_antifabrication-playbook*.md`。
> **一句话事故**：AI 用了一个它「以为存在、实际不存在」的资源当真实文件去读，受挫后还编造理由甩锅用户 / 工具 / 编辑器。这不是一个 bug，是**四个相互独立的失效点串成的链**，必须纵深防御、不能只守一层。

---

## 1. 四层纵深框架

| 层 | 放任会出什么事 | 通用手段（壳） | 时机 | 为何不能只守这一层 |
|----|------|------|------|------|
| **L1 诱因**<br>(工具卡死 / 超时) | 通道异常把 agent 逼向「抄近路」另寻数据源 | ① 排查卡死环境根因；② 卡死后注入降级提示「通道异常 ≠ 资源不存在，禁止自行另找数据源」 | 事前（治理） | 没卡死时 agent 仍可能凭错误世界模型瞎找；治不了已编出来的假动作 |
| **L2 认知**<br>(错误世界模型) | 把「概念上存在」滑坡成「磁盘上必有同名文件」 | 常驻红线 R1：用资源前必先验证存在；读到真实线索必须采纳 | 事前（红线） | 认知对了，应激下仍可能死不认账；运行时拦不住已发出的错误读取 |
| **L3 行为**<br>(死不认账+编造+嫁祸) | 受挫后纯文字编造、甩锅用户 / 工具 / 编辑器 | 常驻红线 R2–R5：验证不到如实说+停下；禁编造；禁甩锅；先认再改 | 事前（红线） | **这条逃逸是 0 工具调用的纯文字输出，任何工具边界 hook 都看不见**，事后无法补救——只能事前约束 |
| **L4 IO/冒头**<br>(真去读不存在的资源) | 唯一肉眼可见的现场：一次注定失败的读取 | 事中**硬拦 deny**（不是 warn），拒绝理由里把「正确资源在哪」投喂回去 | 事中（兜底） | 只守最表层 = 半拉子防线，水从上三层漏；deny 拦得住「读文件」动作，拦不住 L3 的纯文字嫁祸 |

**一句话**：L1/L2/L3 是事前（治诱因 / 治认知 / 治行为），L4 是事中硬兜底。四层各堵不同水路——尤其 L3（纯文字嫁祸）只有事前行为约束能防，IO 层 hook 对它完全失明；L4 的硬 deny 又是 L2 没救住时唯一的实时刹车。**没有任何单层能根治。**

> **白话类比**：像给房子防火——L2「别在屋里堆易燃物」（治认知），L1「电路过载先跳闸提醒」（治诱因），L3「失火了不许谎报是邻居点的」（治应激行为），L4「真冒烟那刻洒水器硬喷」（事中兜底）。少装哪层，火都能从那条缝烧出来。

## 2. 为什么 C3（R1–R5 红线）排最先、且是唯一进产品强制层的

纯文字编造 / 嫁祸（L3）是 0 工具调用的应激输出，**任何工具边界 hook 都看不见、拦不住**，只能事前常驻约束预防、**无事后补救**。所以它：

- **最便宜**（0 代码，纯文字红线）、**最高即时收益**（立刻拿走事故里最伤人的「嫁祸+编造」那段）。
- **零误伤**（不阻断任何工具）→ 唯一够格进 `templates/`「clone 即复印全下游 + 必须 dogfood」强制层的。

条文本身见 rule 文件，本文不重抄（单一事实源）。三个 hook（C1/C2/Stop）各自栽在不同一条上，**都不进 `templates/hooks/`**，设计留本文 + `docs/examples/`，下游撞上痛点再自取落地。下面逐个记决策。

## 3. C1 `deny-nonexistent-resource` 设计（L4，**不进产品层，留 example**）

**定位**：agent 真去读「能解析出路径、且确证不存在」的资源那一刻，事中钩子返回 **deny（硬拦）**，并把「正确资源在哪」写进拒绝理由投喂回去。deny 不是 warn——agent 连系统硬 `FileNotFoundError` 都刹不住，更软的注入式提醒只会更没用。

**四 gate 漏斗（全过才 deny，缺一即放行——刻意做窄交集「宁漏勿误」）**：

| Gate | 判定 | 为何这样设计 |
|------|------|--------------|
| G0 工具匹配 | 命中「读取类工具白名单」 | 只管读资源动作，写 / 执行不碰 |
| G1 读取意图 | 取出目标路径，非空且是「读一个具体文件」（非 glob / 非目录 / 非 URL） | 排除探索式列目录 / 模式匹配等本就允许 miss 的动作 |
| G2 路径可解析 | 能规范成确定本地绝对路径（解析 `~` / 相对拼 cwd / 规范化）；含变量 / 通配解析不出 → 放行 | 解析不出就无法「确证不存在」，证据不足不拦 |
| G3 确证不存在 | 规范化后路径存在性检查 == **False**（确证，不是「我猜」） | 整条防线的证据锚点：必须有「磁盘上真没有」硬证据才动手 |
| G4 豁免 | 不在「豁免前缀集合」内（项目侧配置） | 给「故意读不存在文件是合法行为」留逃生口（touch-before-write / 探针） |

**deny 拒绝理由（两段拼）**：通用前缀（骨架内置）「该路径确证不存在。通道 / 路径错误 ≠ 资源缺失，不要凭空另寻或编造数据源」+ 项目填充 `REAL_SOURCE_HINT`（投喂「这类数据在本项目真实位置」）。

**项目要填的参数（全是数据，不是代码分支）**：`EXEMPT_PREFIXES`（豁免前缀，空列表也能跑）、`REAL_SOURCE_HINT`（真实数据源文案，骨架给通用兜底）、`READ_TOOLS`（可选，读取类工具名集）。

**为什么连休眠骨架都不进 `templates/hooks/`**（两轮 debate 结论）：

1. **C1 唯一有价值的零件 `REAL_SOURCE_HINT` 是项目占位、在 templates 里恒为空**。空 hint 的 C1 ≈ 一条更啰嗦的 `FileNotFoundError`——而模型已证明会无视它。C1 强于系统硬错误**完全条件依赖于 hint 被填**。
2. **C1 误伤是硬停**：一次合法读取被 deny，下游拿到通用拒绝文案、难自诊断、带内不可恢复。rule 误伤只是噪声（模型可折扣），C1 误伤是事故。
3. **dogfood 先伤自己**：bridgeforge 日常高频 touch-before-write、读未创建的 `templates/**` 占位，正命中 C1 的「读取意图明确 ∧ 路径可解析 ∧ 确证不存在」——active-by-default 会先在上游开发流误伤。
4. **休眠代码也复印全下游**：「填俩配置就激活」是危险开关，邀请不理解误伤风险的下游打开 = 维护负担 + 误启用风险，两头不讨好（与 Stop hook 同标准）。

→ **出厂态**：四-gate 判定逻辑做成宿主无关的参考脚本放 [`docs/examples/antifab-deny-hook.py`](examples/antifab-deny-hook.py)，下游真有「幻觉读文件」痛点、又愿填 hint，再照着落地。常驻 rule 的口诀里挂指针导向它，避免「记得去上游翻」的发现性缺口。

## 4. C2 `tool-timeout-governance` 设计（L1，**降级 docs 方法论**）

**机制壳**：边界式超时哨兵——`PreToolUse` 打点起始、`PostToolUse` 算 delta，超阈值 `STALL_THRESHOLD_MS` 判卡死、落结构化日志；卡死后注入固定降级措辞「通道异常 ≠ 资源不存在；工具卡死只说明这次没读成，不代表数据不在那里。禁止据此自行另寻 / 编造数据源，请如实报告『读取超时』并停下问」。项目填 `STALL_THRESHOLD_MS` / 日志落点。

**时机辨正（修正一处常见误解）**：LLM 在工具执行期间是**挂起的、没在生成**，抄近路决策发生在工具**返回之后**那个生成回合。所以 `PostToolUse` 在软超时返回后注入提示，**时机是准的**，不是「事后太晚」。

**但仍降级 docs、不进 `templates/hooks/`**，站得住的理由：

1. **对真·永不返回的硬卡死完全失明**——`PostToolUse` 要工具返回才触发，工具真死就不触发，哨兵瞎掉。而**硬卡死才是最毒的诱因**。
2. **软超时下与 C3 高度重叠**——模型已在上下文看到超时错误，C2 再补一句边际增益薄。
3. **阈值式 delta 检测会误刷噪声**——bridgeforge 大量「慢但成功」的操作（批量铺设）会被误判「疑似超时」；改成「按超时错误签名触发」可消噪，但那已退化成「软超时后再喊一句 C3」，更不值一个强制层 hook + dogfood。

→ **净判**：C2 有**薄而真**的价值（软超时返回后重注入、对冲 C3 长会话衰减），但够不上 `templates/hooks/` + dogfood 门槛 → 降级本文方法论，需要的下游自行实现。

## 5. Stop hook 甩锅自检（L3 实验性硬化，**不进 templates、只留备选**）

**定位**：唯一能**机检纯文字嫁祸**的事后路径——`Stop` hook 扫本轮输出，发现「归因 / 资源名在当次会话真实记录里指不到原文」（R3/R4 的可机判判据）则 `block` 打回。

**为什么不进 `templates/` 代码（哪怕默认关）**：① 准确判定需每回合 LLM judge，**贵**；② 检测难（文字判定）；③ 复印工厂里休眠代码也复印全下游 + 邀请误启用，与 C1 同理。

→ **出厂态**：只把「这是唯一能机检纯文字嫁祸但贵、需 LLM judge」作为已知未来备选记在此，要时再从设计落地，不先囤一坨没人敢开的代码。

## 6. 通用壳 vs 项目填充切分判据

每招都有【通用壳】和【项目填充】。问三个问题，**任一为「是」→ 通用壳（反哺上游）**；**全「否」→ 项目填充（留本地占位）**：

| 判据 | 通用壳 | 项目填充 |
|------|------|------|
| 换个项目还成立吗？ | 原则 / 红线措辞不含项目名词 | 含具体路径 / 文件名 / 接口 / 字段 |
| 是骨架还是血肉？ | 框架结构、hook 判定骨架、规则条目 | 填进骨架的常量：哪个目录豁免、哪条通道会卡、真实数据源在哪 |
| 去掉它别的项目会缺一块吗？ | 是 | 否（只本项目数据布局用得上） |

**一刀切**：措辞出现专有名词（路径 / 文件名 / 接口 / 账户 / 字段 / 运行时目录名）= 项目填充；只剩通用动词和红线 = 通用壳。机制与数据**物理分离**：判定逻辑写死在脚本里（跨项目字节级相同），项目专属只剩**值**（`EXEMPT_PREFIXES` / `REAL_SOURCE_HINT` / `STALL_THRESHOLD_MS`）放项目侧配置。**空配置可降级**：填配置是增强不是前提，下游不填也要能空转不报错、不误伤。

## 7. 落地次序 + 残余风险

**次序（从便宜到贵）**：C3（L2+L3，0 代码红线，最先）→ C2（L1，方法论）→ C1（L4，example，最后）。

**残余风险（诚实标注，不假装 hook 能兜住）**：

1. **纯文字逃逸（最高残余）**：C1/C2/Stop 全收进 docs 后，产品强制层只剩 C3 一道。L3 嫁祸是 0 工具调用，**任何 hook 都看不见** → 完全靠 C3 口诀 + R5 反激励（认错不扣分、掩盖才扣分）压低概率，无事后补救。这是本方案最薄弱、最依赖模型自觉的一环。
2. **C3 概率遵守、长会话衰减**：一行口诀常驻对冲一部分；衰减严重再仿 focus 上「攒够轮数才注入」的节流式信号 hook（暂不做）。
3. **C1 窄交集是「宁漏勿误」**：只拦「确证不存在」，放过「读错了对的文件」「网络盘延迟期的真存在文件」——为零误伤付的代价。
4. **C1/C2 下游自取**：撞上痛点的下游需自行从本文 + example 落地并填项目数据——这是「不污染全下游」换来的代价。

> **接受这个取舍**：宁可少装会误伤 / 够不到的 hook，也不在复印工厂强推半残防御。
