# 双Agent辩论模式

## 流程

### Step 1: 准备
- 在 `doc/2_pending/` 下创建讨论记录文件，命名格式：`debates_YYYY-MM-DD_<简短主题>.md`
- 先用 Explore agent 研读相关代码，建立对问题的充分理解

### Step 2: 多轮辩论（2-3轮）

每轮启动 **两个并行 Agent**：

**Agent A — 提议者（架构师视角）**
- 分析问题根因
- 提出具体解决方案（含代码层面的修改建议）
- 评估方案的优缺点和风险

**Agent B — 审阅者（质疑者视角）**
- 审阅 Agent A 的分析和方案
- 找出逻辑漏洞、遗漏的边界条件、潜在副作用
- 提出替代方案或改进建议

每轮结束后，将双方观点写入讨论记录 MD 文件。
下一轮中，Agent A 需要回应 Agent B 的质疑并修正方案，Agent B 再次审阅。

### Step 3: 收敛共识
当双方观点趋于一致时（通常2-3轮），总结：
- **根因分析**：问题的本质原因
- **推荐方案**：最终达成一致的解决方案
- **实施要点**：关键修改点和注意事项
- **风险提示**：需要额外关注的风险

### Step 4: 确认推进
将最终共识展示给用户，等待用户确认后再开始编码实施。

## 讨论记录格式

```markdown
# 讨论：{主题}
> 日期：{YYYY-MM-DD}
> 相关文件：{文件路径列表}
> 问题描述：{用户原始描述}

## Round 1

### 🔵 提议者
{Agent A 的分析和方案}

### 🔴 审阅者
{Agent B 的审阅意见}

## Round 2

### 🔵 提议者
{Agent A 回应质疑并修正}

### 🔴 审阅者
{Agent B 再次审阅}

## （如需 Round 3 ...）

## ✅ 共识结论
- **根因**：...
- **方案**：...
- **实施要点**：...
- **风险**：...
```

## 重要规则

1. 两个 Agent 必须持**不同立场**，不能互相附和，要有实质性的交锋
2. 每轮讨论都必须基于对实际代码的阅读，不能空谈
3. Agent B 至少要提出一个 Agent A 没有考虑到的问题
4. 最终方案必须包含具体的代码修改建议（文件、函数、改法）
5. 不要超过3轮，如果3轮后仍无法达成一致，列出分歧点交由用户裁决
6. 每轮讨论结果实时写入 MD 文件，确保过程可追溯
