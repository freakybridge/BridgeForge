# 双Agent辩论模式

> 适用：疑难 bug、设计选型、方案风险评估等「需要多视角交锋」的问题。
> **不适合**不需要多视角交锋的小改动；用户在 `confirm` 中选择 `debate` 后才进入本流程。

## 总原则：先确认需求，再辩论方案

辩论本身（多轮 × 双 Agent 并行）是 token 大头。需求、范围、约束和验收统一由 `confirm` 在研读前确认；本 skill 只研读、辩论和确认最终方案，避免重复访谈。

```
confirm → 研读(必要时补事实) → 辩论 → 最终方案确认 → 实施
```

---

### Step 1: 确认卡硬闸

- 先检查本轮是否有 `confirm` 生成、且目标 / 范围 / 约束 / 验收均匹配的确认卡。
- 没有、任务不匹配或已过时：立即调用 `confirm`，在它返回确认卡前禁止研读、创建讨论记录或 spawn agent。
- 有效卡存在：读取它，不得重复需求访谈，进入 Step 2。

### Step 2: 研读（必要时补事实）

- 在 `doc/2_pending/` 创建讨论记录文件，命名格式：`debates_YYYY-MM-DD_<简短主题>.md`，把确认卡路径和已确认目标 / 边界 / 约束写入文件头部
- 用 Explore agent 研读相关代码，建立对问题的充分理解
- 研读中若发现事实缺口，只索取一项必要材料；若发现需求卡不匹配当前任务，返回 `confirm`，别带着错误假设进辩论

### Step 3: 多轮辩论（2-3轮）

**轮次上下文机制（硬性约定，消除歧义）**：
- **第一轮**：spawn 两个并行 Agent（提议者 A、审阅者 B）。**从每个 Agent 工具的返回结果里取出它的 agent id / name**（harness 会在 spawn 返回里标注续接所需的标识），**立刻记入讨论 MD 头部**（如 `> Agent A id: xxx / Agent B id: yyy`），供后续轮次复用。
- **后续轮次**：用 **SendMessage（to: 上面记下的 id）续接同一个 A、B**，**不要**每轮新起 Agent —— 续接能保留它们的完整推理上下文，质量更高也更省 token。
- **降级**：若某轮 SendMessage 续接失败（agent 已回收 / id 失效），则**以 MD 已归档的历轮观点作为上下文，新 spawn 一个同角色 Agent 接续**，并在 MD 记下新 id。这是异常兜底，不与「MD 仅作归档」冲突。
- 正常路径下 MD 只作**可追溯归档**，不承担上下文传递职责（不靠让新 Agent 重读 MD 重建上下文）。

每轮两个 Agent 的角色：

**Agent A — 提议者（架构师视角）**
- 分析问题根因
- 提出具体解决方案（含代码层面的修改建议）
- 评估方案的优缺点和风险

**Agent B — 审阅者（质疑者视角）**
- 审阅 Agent A 的分析和方案
- 找出逻辑漏洞、遗漏的边界条件、潜在副作用
- 提出替代方案或改进建议

每轮结束后把双方观点写入 MD。下一轮 A 回应 B 的质疑并修正方案，B 再次审阅。

### Step 4: 收敛方案大纲与最终确认

双方趋于一致时（通常 2-3 轮），**先给一个简短方案大纲**（不甩长文档），各一两句：
- **根因** / **推荐方案** / **实施要点** / **风险**

> ⛔ **硬契约**：把方案大纲作为一个选择题交给用户确认或纠正，本步以该题结束回合；用户未拍板前禁止开始任何编码。

### Step 5: 实施

用户确认后才开始编码。若实施本身是涉及多文件的大改动，**建议顺势转 `/collab`（Claude）/ `$collab`（Codex）分治执行**。

## 讨论记录格式

```markdown
# 讨论：{主题}
> 日期：{YYYY-MM-DD} / 相关文件：{...} / 确认卡：{路径} / 目标·范围·约束：{确认卡摘要} / 问题：{用户原始描述}

## Round 1
### 🔵 提议者 {A 的分析和方案}
### 🔴 审阅者 {B 的审阅意见}

## Round 2（🔵 A 回应质疑并修正 / 🔴 B 再审；如需 Round 3 同式）

## ✅ 共识结论
- **根因** / **方案** / **实施要点（文件·函数·改法）** / **风险**：...
```

## 重要规则

1. 两个 Agent 必须持**不同立场**，不能互相附和，要有实质性的交锋
2. 每轮讨论都必须基于对实际代码的阅读，不能空谈
3. Agent B 至少要提出一个 Agent A 没有考虑到的问题
4. 最终方案必须包含具体的代码修改建议（文件、函数、改法）
5. 不要超过 3 轮，如果 3 轮后仍无法达成一致，列出分歧点交由用户裁决
6. **轮次间用 SendMessage 续接同一 A / B**，MD 仅作归档（每轮结果实时写入），不靠重读 MD 传递上下文
7. **确认卡硬闸**：没有有效 `confirm` 需求卡，禁止研读、创建记录或启动辩论 Agent
8. **最终确认**：方案大纲必须以一个选择题获得确认，才可实施
