# debate — 双 agent 辩论

## 定位与边界

对需要多视角交锋的问题形成可实施共识。需求范围由 `confirm` 统一确认；本 skill 只负责研读、辩论、方案确认和后续实施交接。

## 输入

- 与当前目标、范围、约束和验收匹配的确认卡。
- `$ARGUMENTS`：问题现象、相关模块、已尝试方案和证据。
- 相关真实代码、规范和测试入口。

## 核心流程

### 1. 确认卡硬闸

没有有效确认卡、卡片不匹配或已过时，就调用 `confirm`；返回前禁止研读、创建讨论记录或启动 agent。有效卡存在时直接读取，不重复访谈。

### 2. 研读与建档

1. 在 `doc/2_pending/` 创建 `debates_YYYY-MM-DD_<topic>.md`，记录确认卡路径、目标、边界、约束和问题。
2. 把 read-only research 显式分派给 `light-explorer`，研读真实代码和证据；只在必要事实缺失时索取一项材料。发现确认卡不匹配就返回 `confirm`。

### 3. 进行 2-3 轮辩论

1. 第一轮并行启动两个 agent，并立即把返回的 agent id / name 写入讨论记录：
   - A 使用 `implementation-worker`：分析根因，提出代码级方案并评估取舍与风险。
   - B 使用 `review-auditor`：挑战 A 的假设，找边界、副作用和替代方案；至少提出一个 A 未覆盖的问题。
2. 后续轮次必须向同一 A、B 发送消息，保留上下文；只有续接失败时，才让新 agent 读取记录后接替，并更新 id。
3. 每轮都基于实际代码，实时记录双方观点。A 回应质疑并修正，B 再审；最多 3 轮。
4. 两个角色必须实质交锋，不得互相附和。

### 4. 收敛与确认硬闸

给出根因、推荐方案、实施要点和风险的短大纲，并把接受、调整或放弃作为一个选择题。当前回合在该题结束；用户未确认前禁止编码。

3 轮后仍有分歧时，列出分歧和证据，让用户裁决，不继续增加轮次。

### 5. 实施交接

用户确认后按方案实施；若涉及多文件且可并行，转入 `collab`。不得把未经确认的辩论结论直接落地。

## 输出与收据

- 讨论记录路径和 A / B agent 标识。
- 每轮关键主张、挑战、证据与方案变化。
- 共识或分歧：根因、文件 / 函数级改法、风险和待验证项。
- 用户最终选择及后续实施入口。

## 停止条件

- 无有效确认卡、关键事实缺失或用户未确认最终大纲时停止。
- 第 3 轮仍未收敛时交用户裁决。
- agent 续接失败时记录降级事实；无法可靠重建上下文就停止。

## 禁止事项

- 禁止辩论未读真实代码的抽象观点。
- 禁止每轮新建 A / B 丢失上下文，除非原 agent 已失效。
- 禁止超过 3 轮或让两个 agent 采取相同立场。
- 禁止用户确认最终大纲前开始编码。
- 禁止没有有效确认卡就研读或启动辩论。
