# 双Agent辩论模式

> 适用：疑难 bug、设计选型、方案风险评估等「需要多视角交锋」的问题。
> **不适合**已经清楚怎么做的小改动 —— 那种直接做，别套重型流程（见 Step 1 轻量出口）。

## 总原则：渐进对齐，每道闸都停下等用户

辩论本身（多轮 × 双 Agent 并行）是 token 大头。所以流程被切成几道「闸」，每道闸都卡在烧 token 的动作**之前**停下等你拍板，方向一偏立即纠正，绝不带着错误目标全程空烧。

```
闸① 引导发问 → 闸② 复述确认 → 研读(可追问) → 辩论 → 闸③ 方案大纲 → 实施
```

---

### Step 1: Welcome + 边界 + 引导发问（闸①，研读前）

先向用户**简述**：本模式会做什么、适用边界、大致会起多少 Agent / token 量级。然后用 **AskUserQuestion** 抛出引导性问题对齐目标：

1. **期望产出**：定位根因 / 方案选型并拍板 / 评估某个既定方案的风险？
2. **讨论边界**：限定哪个模块、哪几个文件？哪些明确不在范围内？
3. **已排除方向**：你已经试过或否决了哪些思路（避免两个 Agent 重走老路）？

> ⛔ **硬契约**：本步必须以 **AskUserQuestion 调用结束当前回合**，用户未回答前**禁止**研读 / spawn 任何 agent / 往下走任何一步。

> **轻量出口**：若用户回答后发现问题其实很小、不够格用辩论（粗判据：预计 ≤2 文件、单线即可定位），**直接说「这个我直接做就行，不必起辩论」并退出**，不硬走完流程。

### Step 2: 复述确认（闸②）

用一两句话**复述**你理解的辩论目标 / 边界 / 已排除方向，作为 **AskUserQuestion 抛给用户**（「我理解的目标/边界是 X，对吗？」）。

> ⛔ **硬契约**：本步必须以 **AskUserQuestion 调用结束当前回合**，**不得**在同一回合里复述完就继续研读。用户点头前**禁止进入 Step 3**。闸①②虽相邻，也**绝不可合并**为一回合连跑。

### Step 3: 研读（可追加发问）

- 在 `doc/2_pending/` 创建讨论记录文件，命名格式：`debates_YYYY-MM-DD_<简短主题>.md`，把闸①② 对齐到的目标 / 边界 / 已排除方向写入文件头部
- 用 Explore agent 研读相关代码，建立对问题的充分理解
- 研读中若发现关键信息缺口（某模块行为不清、缺背景），**立即用 AskUserQuestion 追问**，别带着错误假设进辩论

### Step 4: 多轮辩论（2-3轮）

**轮次上下文机制（硬性约定，消除歧义）**：
- **第一轮**：spawn 两个并行 Agent（提议者 A、审阅者 B）。**从每个 Agent 工具的返回结果里取出它的 agent id / name**（harness 会在 spawn 返回里标注续接所需的标识），**立刻记入讨论 MD 头部**（如 `> Agent A id: xxx / Agent B id: yyy`），供后续轮次复用。
- **后续轮次**：用 **SendMessage（to: 上面记下的 id）续接同一个 A、B**，**不要**每轮新起 Agent —— 续接能保留它们的完整推理上下文，质量更高也更省 token。
- **降级**：若某轮 SendMessage 续接失败（agent 已回收 / id 失效），则**以 MD 已归档的历轮观点作为上下文，新 spawn 一个同角色 Agent 接续**，并在 MD 记下新 id。这是异常兜底，不与「MD 仅作归档」冲突。
- 正常路径下 MD 只作**可追溯归档**，不承担上下文传递职责（不靠让新 Agent 重读 MD 重建上下文）。

每轮两个 Agent 的角色：

**Agent A — 提议者（架构师视角）**
- 分析问题根因
- 提出具体解决方案（含代码层面的修改建议）
- 评估方案的优缺点和风险

**Agent B — 审阅者（质疑者视角）**
- 审阅 Agent A 的分析和方案
- 找出逻辑漏洞、遗漏的边界条件、潜在副作用
- 提出替代方案或改进建议

每轮结束后把双方观点写入 MD。下一轮 A 回应 B 的质疑并修正方案，B 再次审阅。

### Step 5: 收敛方案大纲（闸③）

双方趋于一致时（通常 2-3 轮），**先给一个简短方案大纲**（不甩长文档），各一两句：
- **根因** / **推荐方案** / **实施要点** / **风险**

> ⛔ **硬契约**：把方案大纲作为 **AskUserQuestion 抛给用户确认或纠正**，本步以该工具调用**结束回合**；用户未拍板前**禁止开始任何编码**。

### Step 6: 实施

用户确认后才开始编码。若实施本身是涉及多文件的大改动，**建议顺势转 `/collab` 分治执行**。

## 讨论记录格式

```markdown
# 讨论：{主题}
> 日期：{YYYY-MM-DD}
> 相关文件：{文件路径列表}
> 辩论目标 / 边界 / 已排除方向：{闸①② 对齐结果}
> 问题描述：{用户原始描述}

## Round 1

### 🔵 提议者
{Agent A 的分析和方案}

### 🔴 审阅者
{Agent B 的审阅意见}

## Round 2

### 🔵 提议者
{Agent A 回应质疑并修正}

### 🔴 审阅者
{Agent B 再次审阅}

## （如需 Round 3 ...）

## ✅ 共识结论
- **根因**：...
- **方案**：...
- **实施要点**：...
- **风险**：...
```

## 重要规则

1. 两个 Agent 必须持**不同立场**，不能互相附和，要有实质性的交锋
2. 每轮讨论都必须基于对实际代码的阅读，不能空谈
3. Agent B 至少要提出一个 Agent A 没有考虑到的问题
4. 最终方案必须包含具体的代码修改建议（文件、函数、改法）
5. 不要超过 3 轮，如果 3 轮后仍无法达成一致，列出分歧点交由用户裁决
6. 每轮讨论结果实时写入 MD 文件，确保过程可追溯
7. **轮次间用 SendMessage 续接同一 A / B**，MD 仅作归档，不靠重读 MD 传递上下文
8. **三道闸（引导发问 / 复述确认 / 方案大纲）必须各自停下等用户**，不得连贯跑完
9. **轻量出口**：对齐 / 复述阶段若发现任务太小不够格辩论，提前退出并建议直接做
