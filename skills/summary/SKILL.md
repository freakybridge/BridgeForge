# 对话总结

总结本次对话中的重要决策和经验，完成 memory、rules、docs 的同步更新。

## 步骤

### 1. 回顾本次对话
梳理以下内容：
- **架构决策**：做了哪些设计选择？为什么？
- **踩过的坑**：遇到了什么 bug 或误解？根因是什么？
- **已完成的功能**：实现了哪些模块/功能？
- **遗留问题**：未解决的问题或下次需要继续的工作

### 2. 更新 memory
将值得跨对话保留的内容写入对应 memory 文件：
- 架构决策 → `project` 类型 memory
- 踩坑经验（防止我重犯）→ `feedback` 类型 memory
- 新的外部资源/工具 → `reference` 类型 memory

memory 目录：`.claude/memory/`（项目内 junction 透明指向系统路径）
同时更新 `MEMORY.md` 索引。

### 3. 检查 rules 是否需要更新
对照 `.claude/rules/` 下的规则文件，检查本次对话中是否发现：
- 新的反复出现的模式或约定（应提炼为规则）
- 踩坑暴露出 rules 中尚未覆盖的禁止事项
- 已有规则需要修正或补充的内容

如有需要，更新对应的 `rules/*.md` 文件（参见 `CLAUDE.md` §2 规则索引确定写入位置）。

### 4. 检查相关文档是否需要同步
对照 `rules/workflow.md` §1 的文档同步表，检查本次改动是否涉及需要同步的文档（架构红线变更 / 接口契约 / 新增数据对象等）。

如有需要，执行 `/sync-docs` 或直接更新对应文档。

### 5. 清理已解决的 TODO 和 current 文档（防止无限膨胀）

扫描 `doc/0_architecture/TODO-INDEX.md` 和 `doc/2_pending/` 下的活跃 .md，找出本次对话真正解决的条目。

**判定标准（严格，必须同时满足）**：
1. 代码已合并（本次对话中实际 edit 了文件并 build 通过）
2. 用户已**显式确认**解决（实操验证 / 看过 diff / 明确说"OK" / 测试通过）

**禁止**把"代码已写但未验证"的条目算作解决。例：今天刚改完几条 TODO 但用户没实操验证，不能算。

**执行流程**（禁止自动批量删）：

1. **列候选清单**给用户，分两类展示：

   ```
   候选清理（TODO-INDEX 删行）:
   - [ ] #XX 一句话摘要 — 本次做了什么
   - [ ] #YY ...

   候选归档（current/ → archive/）:
   - [ ] doc/2_pending/2026-XX-XX_foo.md — 对应的已完成工作
   ```

2. **等用户逐条 y/n 或整体确认**。用户说"都清"也行，但必须显式。

3. **执行清理**（按用户勾选）：
   - **TODO-INDEX 条目** → 直接 `Edit` 删整行，不保留 strikethrough
   - **doc/2_pending/*.md** → `git mv doc/2_pending/XXX.md doc/4_archive/XXX.md` 保留决策 trail，禁止 `rm`
   - **doc/README.md** 索引 → 从 `## current/` 表格里删对应行，如归档则在 `## archive/` 表格加新行

4. **更新 TODO-INDEX 短期 TODO 条目数**（顶部 `> 短期 TODO 条目：N`）。

**兜底**：若本次对话**没有**任何条目满足判定标准（常见情况），跳过本步，向用户说明"本次未达清理标准的条目"即可，不强行凑数。

### 6. 反哺候选捕捉（通用经验回流 setup_agent 上游）

回顾本次写入的 memory / rules：**有没有哪条经验换个项目也成立**（不依赖本项目业务领域）？有 → 它是 setup_agent 框架的**反哺候选**，记一行进收件箱，等以后 `/harvest` 脱敏推回上游骨架库。

**判定"通用"**（参考 `.claude/rules/meta_rule_design.md`「模板 vs 占位」精神）：
- ✅ 通用：协作工作流 / 调试方法论 / 可移植性坑 / agent 行为约定 —— 跟本项目业务无关，换项目照样成立
- ❌ 不通用：具体业务术语 / 某语言某库的专属坑 / 内部包名·路径·凭证

**操作（命中才做，否则跳过本步）**：
1. 向用户确认："这条像是通用的（跟本项目业务无关），记进反哺收件箱？"
2. 用户点头 → 追加一行到 `.claude/harvest-inbox.md`（无则新建），格式：
   ```
   - [ ] <一句话:学到啥> | 来源 <file:line 或 本轮对话> | 通用性 <为啥跨项目成立> | <日期>
   ```

> 这里只**捕捉**，不推送。推送由 `/harvest` 在另一时刻做（读收件箱 → 脱敏 → 写上游 templates）。
> 急用：一眼就确定要立即反哺的，不必等收件箱攒够——直接 `/harvest <这条>` 走快车道。

### 7. 输出摘要
向用户展示：
- 写入了哪些 memory（类型 + 一句话描述）
- 更新了哪些 rules（如有）
- 需要同步的文档（如有）
- **清理了哪些 TODO / 归档了哪些 current 文档（如有）**
- **记了哪些反哺候选进 `.claude/harvest-inbox.md`（如有）**
- 有哪些遗留问题需要下次继续

## 规则
- memory 只写跨对话有价值的内容，当前对话的临时状态不写
- 不重复已有的 memory，先检查再决定是新建还是更新
- rules 和 docs 的检查以"是否影响其他模块开发者做决策"为判断标准
