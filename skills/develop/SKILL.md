# develop — 大需求交付流水线

## 定位与边界

串联“有效确认卡 → 实施 → 独立验证 → 用户试用”。需求访谈由 `confirm` 完成；本 skill 不重复确认业务细节。单文件小修、文案微调、明确 bug 的一次性修复、纯咨询或只读方案不进入本流程。

## 输入

- 与当前目标、范围、约束和验收匹配的 `confirm` 需求卡。
- `$ARGUMENTS`：用户的大需求描述。
- 项目真实代码、文档规范、测试入口和 Git 状态。

## 核心流程

### 1. 开工硬闸

1. 检查本轮有效确认卡；没有、不匹配或已过时就调用 `confirm`，在它返回前禁止研读或开发。
2. 读取有效卡，不重复访谈，只问：

```text
问题：是否按这份已确认需求开始开发？
A. 是，开始落盘并实现
B. 否，回到 confirm 修正需求
C. 暂不实现
```

3. 只有用户明确选择 A 或给出唯一对应 A 的无条件回答，才进入实施。

### 2. 完善需求包

1. 在确认卡中补充实施计划、实施记录和验证记录，不重建第二份需求。
2. 项目级长期约束更新 `doc/0_architecture/`；单 feature 保持在 `doc/1_plan/<topic>/`；暂缓项进入 TODO 或 `doc/2_pending/`。
3. 需求包至少保留背景与目标、非目标、用户可见行为、约束与风险、验收、暂缓项和实施假设。
4. 新增、删除或重命名 `doc/**.md` 时同步 `doc/README.md`。
5. 此后自动推进，只在卡点、高风险或完成时打断用户。

### 3. 拆解与实现

1. 把 read-only discovery 显式分派给 `light-explorer`，让它读取代码和文档并返回单一事实源、既有模式和风险收据；主 agent 据此决策。
2. 预计不超过 2 个文件或依赖强时单线执行；文件边界清晰且无依赖时并行。
3. 实现阶段显式分派给 `implementation-worker`。并行前定义接口和文件边界；同组 agent 禁止改同一文件。每个任务必须携带整体目标、需求卡路径、验收、边界和接口。
4. 子任务最多重试 1 次；再次失败即报告卡点，不循环。
5. 需要并行时复用 `collab` 的执行约束，但不得重新走其用户确认闸。

### 4. 同步事实与文档

- 已确认范围内的实现细节、事实补全和验证状态直接更新需求包或设计文档。
- 任何用户行为、业务规则、接口、范围或验收变化都必须暂停，以单题选择确认；不得用文档追认未确认变化。
- 完成后更新实施状态、变更记录、每项验收状态及相关设计 / rules。

### 5. 独立验证

1. 运行相关 lint、类型检查、单测、集成测试或手工验证脚本。
2. 把 delivery review 显式分派给 `review-auditor`，向它提供需求卡、验收、改动文件，并要求独立读取真实文件和 `git diff`，核对遗漏、串联、回归和文档。
3. 规则和可执行测试优先于 LLM review；平台无独立 agent 能力时，明确标注未完成该验证。

### 6. 试用闭环

1. 交付改动、需求卡、验证收据、用户试用主路径和剩余风险。
2. 属于当前需求的小 bug 直接修并更新同一需求包；新范围进入 TODO 或新的 `develop`。
3. 同一症状修复失败两次，停止并按项目鬼打墙规则升级。

## 输出与收据

- 需求卡与相关设计文档路径。
- 实际改动文件及用户可见行为。
- 每条验证命令、断言、覆盖场景和独立 review 结论。
- 未通过、未验证项及试用主路径。

## 停止条件

- 开工选择题、需求边界变化或高风险操作出现时，结束当前回合等待用户。
- 缺少关键输入、子任务连续两次失败或验证无法继续时，报告卡点。
- 独立验证不可用时允许交付，但必须明确标为未完成。

## 禁止事项

- 禁止无有效确认卡开发或重复 `confirm` 访谈。
- 禁止用户确认开工前改代码或配置。
- 禁止 Step 2 后把普通实现细节反复交回用户决定。
- 禁止用主 agent 总结代替独立读取真实 diff。
- 禁止遗漏 `doc/README.md` 索引同步。
- 禁止自动 `git add`、commit 或 push。
