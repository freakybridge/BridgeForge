# develop — 大需求交付流水线

定位：把“统一确认需求 → 落盘实现 → 独立验证 → 用户试用反馈”串成默认闭环。需求确认统一由 `confirm` 完成；本 skill 只负责开发、验证和试用反馈。

## 适用边界

适用：
- 新功能 / 大需求 / 跨模块改动
- 验收口径尚不清楚，先要讨论需求
- 用户明确说“按完整流程做”“开一个需求”“走 develop”
- 小改动看似简单，但会改变多个模块、文档或用户工作流

不适用：
- 单文件小修、文案、样式微调、明确 bug 的一次性修复
- 纯咨询 / 评估 / 方案判断，用户还没表达执行意图
- 只需要 `/plan` 只读分析或 `/debate` 方案辩论的任务

## 工作流

### Step 1：读取确认卡并确认实施

- 先检查本轮是否已有 `confirm` 生成、且目标 / 范围 / 约束 / 验收均匹配的确认卡。
- 没有、任务不匹配或已过时：立即调用 `confirm`，在它返回确认卡前不得研读、落盘或开发。
- 有效卡存在：读取它，不得重复需求访谈；然后只问一个选择题：

```text
问题：是否按这份已确认需求开始开发？
A. 是，开始落盘并实现
B. 否，回到 confirm 修正需求
C. 暂不实现
```

只有用户选 A 或作出唯一对应 A 的无条件同义回答，才能进入 Step 2。

### Step 2：落盘需求包

用户确认 Step 1 后，确认卡已由 `confirm` 落盘；本步骤只补充实施计划、实施记录和验证记录，再写代码。Step 2 开始后属于自动执行闭环，不要再为“如何拆任务 / 先改哪个文件”反复打断用户。

文档位置按 `doc/` 规范判断：
- 项目级长期需求 / roadmap / 架构约束：更新 `doc/0_architecture/` 下对应 PRD / roadmap / acceptance 文档。
- 单个活跃 feature：写入 `doc/1_plan/<topic>/requirements_YYYY-MM-DD_<slug>.md`。
- 该 feature 的任务拆解、开发记录、验证记录，继续放在同一个 `doc/1_plan/<topic>/` 子目录。
- 暂缓项写入 `doc/0_architecture/TODO-INDEX.md` 或 `doc/2_pending/`，不要混进当前实施范围。
- 新增 / 删除 / 重命名任何 `doc/**.md` 后，必须同步 `doc/README.md` 唯一索引。

需求包至少包含：
- 背景与目标
- 非目标
- 用户可见行为
- 约束 / 风险边界
- 验收清单
- 暂缓项
- 实施假设

写完需求包后，继续自动推进 Step 3。后续只在三类情况中断用户：
- **卡点**：缺少关键输入，不问就无法继续。
- **高风险**：会覆盖 / 删除 / 迁移重要内容，或实现发现需求边界、验收口径需要改变。
- **完成**：交付结果、验证证据、剩余风险和试用路径。

### Step 3：自动拆解与实现

落盘需求包后，由 agent 自行拆解任务：
- 先读相关代码 / 文档，找单一事实源和既有模式。
- 选择单线还是并行：预计 ≤2 文件或强依赖顺序明显 → 单线；可拆成无依赖子任务且文件边界清楚 → 多 agent 并行。
- 多 agent 并行时，必须先定义接口约定和文件边界；同一并行组内禁止多个 agent 修改同一文件。
- 每个子任务 prompt 必须包含整体目标、需求文档路径、验收清单、文件边界、接口约定。
- 子任务失败最多重试 1 次；仍失败则停下报告卡点，不闷头循环。

需要并行时，复用 `collab` 的执行约束，但不要把用户重新带进 `collab` 的三道闸。`develop` 已经在 Step 1 完成修改方向确认，后续只在卡点、高风险或完成交付时回到用户。

### Step 4：同步文档

实现过程中若发现偏差只涉及已确认范围内的实现细节、事实补全或验证状态，立即更新同一需求包或对应设计文档。若偏差涉及已确认的用户可见行为、业务规则、接口、范围或验收，先暂停并以单题选择确认变更；只有用户确认后，才能更新需求包或设计文档并继续。不得用文档更新追认未经确认的变化。

完成开发后至少更新：
- 需求包的“实施状态 / 变更记录”
- 验收清单的通过 / 未通过 / 未验证状态
- 如有架构或接口变化，对应 `doc/3_design/` 或 rules

### Step 5：交付前独立验证

交付前必须启动独立 verification / review agent；它不能只读主 agent 的总结，必须独立看真实改动。

验证 agent prompt 必须包含：
- 需求文档路径
- 验收清单
- 本次改动文件清单
- 要求它独立 Read 关键文件 / 查看 `git diff`
- 要求它核对遗漏、接口串联、回归风险、文档是否同步

同时运行相关可执行验证（lint / 类型检查 / 单测 / 集成测试 / 手工脚本）。rules-based 验证优先，LLM review 只作补充。

若当前平台没有可用独立 agent 能力，必须在交付中明确写“未完成独立 agent 验证”，不得声称完整交付前验证通过。

### Step 6：用户试用与反馈闭环

交付给用户时只报告：
- 改了什么
- 需求文档在哪里
- 验证了什么，证据是什么
- 用户该试用哪条主路径
- 剩余风险 / 未验证项

用户反馈小 bug 时：
- 先判断是否仍属当前需求闭环。
- 小 bug 直接修，更新同一需求包的反馈记录和验收状态。
- 新范围 / 新功能 / 设计替换，写入 TODO 或开启新的 `develop`，不要悄悄扩张当前需求。
- 同一症状修复失败两次后停下，按项目鬼打墙规则升级。

## 需求包模板

```markdown
# 需求：{标题}
> 日期：{YYYY-MM-DD}
> 状态：draft / implementing / verifying / trial / done
> 入口：{用户原始需求摘要}

## 背景与目标
{为什么做，做成什么}

## 非目标
{明确不做什么}

## 用户可见行为
{用户如何感知它}

## 约束 / 风险边界
{不能破坏的行为、接口、性能、兼容性}

## 验收清单
- [ ] {可验证断言 1}
- [ ] {可验证断言 2}

## 暂缓项
- {不进本轮的事项，必要时同步 TODO-INDEX}

## 实施计划
{单线或子任务拆解；用户无需拍板}

## 实施记录
{开发中持续更新}

## 验证记录
{命令 / receipt / 独立 agent 结论 / 未验证项}

## 用户试用反馈
{反馈、修复、复验记录}
```

## 红线

1. 没有有效 `confirm` 需求卡时，必须转入 `confirm`；不得自行澄清、降级或开发。
2. 不得重复 `confirm` 已完成的需求访谈；只在实施前保留 Step 1 的最终选择题。
3. 未在 Step 1 得到用户开始开发的确认前，禁止改代码或改配置。
4. Step 2 以后自动推进；除卡点、高风险、完成交付外，不把用户拉回开发流程细节。
5. `doc/README.md` 必须同步。
6. 交付前必须独立验证；做不到就明示未完成。
7. 不自动 `git add` / `commit` / `push`，除非用户明确要求。

输入：用户的大需求 / 新功能描述（目标、背景、期望行为、约束、验收线索等）。
