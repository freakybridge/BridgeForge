# 白话解释

## 定位与边界

解释用户给出的内容或当前对话中的对象，不替用户执行该对象所描述的任务。保留原有条件、不确定性和限制，不能把推测改写成事实。

## 输入

- `$ARGUMENTS` 有内容时，以它为主要解释对象，并结合当前话题补足指代。
- `$ARGUMENTS` 为空时，只使用当前对话里能明确定位的内容。
- 无法确定解释对象、缺少原文或缺少必要上下文时，明确指出缺什么，不猜测。

## 核心流程

1. 识别解释对象、用户卡住的点和用户当前请求所用语言。
2. 先提取原内容的结论，再核对成立条件、不确定性和限制。
3. 按以下顺序输出：
   - **结论**：用一句话直接回答“这是什么意思”。
   - **白话解释**：拆开因果或步骤；必须使用术语时，紧跟一句白话释义。
   - **当前话题例子**：用用户正在讨论的对象、角色或数据举例；假设内容必须标明是假设。
4. 使用用户当前请求的语言；中文请求使用简体中文。
5. 输出完成后停止，不顺手实现、修改、重设计或扩大被解释的任务。

## 触发与预期

| 请求 | 预期 |
|------|------|
| `$explain <内容>` | 直接按三段顺序解释指定内容 |
| “白话讲讲”“举个例子”“我没看懂” | 从当前话题定位对象，再按三段顺序解释 |
| 没有可定位的原文或上下文 | 三段结构仍保留，但明确说明缺失内容以及因此无法确认的部分 |

## 输出与验证

- 三段都必须出现，标题跟随用户语言。
- 例子必须能映射回当前话题，不能用无关的万能故事替代解释。
- 输出前复核：有没有漏掉“仅在……时”、可能性、例外或未知项。
- 若原内容本身互相矛盾，指出矛盾位置，不替它强行补成自洽结论。

## 停止条件

- 缺少解释对象或关键上下文：说明缺口后停止，等待用户提供内容。
- 用户实际要求实施、修改或重设计：说明本 skill 只负责解释，不执行该扩展任务。

## 禁止事项

- 禁止把“可能”“通常”“尚未验证”改写成“必然”“总是”“已经确认”。
- 禁止编造原文、代码行为、数据、业务规则或用户意图。
- 禁止为了举例引入会改变结论的新前提。
- 禁止解释完成后自动修改文件、运行实现命令或提出额外设计。

输入：`$ARGUMENTS`
