# feature-dev — 大需求交付流水线

定位：把“讨论需求 → 落盘需求文档 → 自动开发 → 独立验证 → 用户试用反馈”串成一个默认闭环。用户只参与需求边界和试用反馈，不参与任务拆分、agent 调度、实现顺序等开发流程细节。

## 适用边界

适用：
- 新功能 / 大需求 / 跨模块改动
- 验收口径尚不清楚，先要讨论需求
- 用户明确说“按完整流程做”“开一个需求”“走 feature-dev”
- 小改动看似简单，但会改变多个模块、文档或用户工作流

不适用：
- 单文件小修、文案、样式微调、明确 bug 的一次性修复
- 纯咨询 / 评估 / 方案判断，用户还没表达执行意图
- 只需要 `/plan` 只读分析或 `/debate` 方案辩论的任务

轻量出口：研读前若确认预计不超过 2 个文件、验收明确、无需落盘需求文档，说明“这不是大需求，直接单线做”，退出本 skill 并直接执行普通开发流程。

## 工作流

### Step 1：需求澄清

沿用入口文件的 `[clarify]` 渐进澄清规则：
- 每轮最多问一个真正会改变开发路线、验收标准或风险边界的问题。
- 每累计 3 个问题，暂停复述已确认目标、非目标、验收口径、剩余卡点。
- 超过 6 个问题仍未收敛，停止追问，输出 PRD / 验收草案让用户改。
- 当能用 3-5 句话说清目标、非目标、触发条件、验收口径、已确认项 / 合理假设 / 剩余风险时，停止追问。

只问需求问题，不问开发流程问题。不要让用户选择拆分粒度、agent 数量、文件顺序、测试命令组合；这些由 agent 自己判断。

### Step 2：落盘需求包

需求收敛后，先写文档，再写代码。

文档位置按 `doc/` 规范判断：
- 项目级长期需求 / roadmap / 架构约束：更新 `doc/0_architecture/` 下对应 PRD / roadmap / acceptance 文档。
- 单个活跃 feature：写入 `doc/1_plan/<topic>/requirements_YYYY-MM-DD_<slug>.md`。
- 该 feature 的任务拆解、开发记录、验证记录，继续放在同一个 `doc/1_plan/<topic>/` 子目录。
- 暂缓项写入 `doc/0_architecture/TODO-INDEX.md` 或 `doc/2_pending/`，不要混进当前实施范围。
- 新增 / 删除 / 重命名任何 `doc/**.md` 后，必须同步 `doc/README.md` 唯一索引。

需求包至少包含：
- 背景与目标
- 非目标
- 用户可见行为
- 约束 / 风险边界
- 验收清单
- 暂缓项
- 实施假设

写完需求包后，给用户一个短确认：只确认“需求是否写对”。用户确认后，后续开发自动推进；不要再就任务拆分或执行流程反复征求意见。若用户明确说“直接按这个做”，可不额外停一轮。

### Step 3：自动拆解与实现

确认需求后，由 agent 自行拆解任务：
- 先读相关代码 / 文档，找单一事实源和既有模式。
- 选择单线还是并行：预计 ≤2 文件或强依赖顺序明显 → 单线；可拆成无依赖子任务且文件边界清楚 → 多 agent 并行。
- 多 agent 并行时，必须先定义接口约定和文件边界；同一并行组内禁止多个 agent 修改同一文件。
- 每个子任务 prompt 必须包含整体目标、需求文档路径、验收清单、文件边界、接口约定。
- 子任务失败最多重试 1 次；仍失败则停下报告卡点，不闷头循环。

需要并行时，复用 `collab` 的执行约束，但不要把用户重新带进 `collab` 的三道闸。`feature-dev` 已经完成需求确认，后续只在发现会改变需求边界、验收口径或高风险不可逆操作时回问用户。

### Step 4：同步文档

实现过程中若代码行为、接口或范围与需求包不同，立即更新同一需求包或对应设计文档。不要等交付后靠总结补锅。

完成开发后至少更新：
- 需求包的“实施状态 / 变更记录”
- 验收清单的通过 / 未通过 / 未验证状态
- 如有架构或接口变化，对应 `doc/3_design/` 或 rules

### Step 5：交付前独立验证

交付前必须启动独立 verification / review agent；它不能只读主 agent 的总结，必须独立看真实改动。

验证 agent prompt 必须包含：
- 需求文档路径
- 验收清单
- 本次改动文件清单
- 要求它独立 Read 关键文件 / 查看 `git diff`
- 要求它核对遗漏、接口串联、回归风险、文档是否同步

同时运行相关可执行验证（lint / 类型检查 / 单测 / 集成测试 / 手工脚本）。rules-based 验证优先，LLM review 只作补充。

若当前平台没有可用独立 agent 能力，必须在交付中明确写“未完成独立 agent 验证”，不得声称完整交付前验证通过。

### Step 6：用户试用与反馈闭环

交付给用户时只报告：
- 改了什么
- 需求文档在哪里
- 验证了什么，证据是什么
- 用户该试用哪条主路径
- 剩余风险 / 未验证项

用户反馈小 bug 时：
- 先判断是否仍属当前需求闭环。
- 小 bug 直接修，更新同一需求包的反馈记录和验收状态。
- 新范围 / 新功能 / 设计替换，写入 TODO 或开启新的 `feature-dev`，不要悄悄扩张当前需求。
- 同一症状修复失败两次后停下，按项目鬼打墙规则升级。

## 需求包模板

```markdown
# 需求：{标题}
> 日期：{YYYY-MM-DD}
> 状态：draft / implementing / verifying / trial / done
> 入口：{用户原始需求摘要}

## 背景与目标
{为什么做，做成什么}

## 非目标
{明确不做什么}

## 用户可见行为
{用户如何感知它}

## 约束 / 风险边界
{不能破坏的行为、接口、性能、兼容性}

## 验收清单
- [ ] {可验证断言 1}
- [ ] {可验证断言 2}

## 暂缓项
- {不进本轮的事项，必要时同步 TODO-INDEX}

## 实施计划
{单线或子任务拆解；用户无需拍板}

## 实施记录
{开发中持续更新}

## 验证记录
{命令 / receipt / 独立 agent 结论 / 未验证项}

## 用户试用反馈
{反馈、修复、复验记录}
```

## 红线

1. 先落盘需求包，再开发；除非确认是轻量出口。
2. 只向用户确认需求边界，不让用户管理开发流程。
3. `doc/README.md` 必须同步。
4. 交付前必须独立验证；做不到就明示未完成。
5. 不自动 `git add` / `commit` / `push`，除非用户明确要求。

输入：用户的大需求 / 新功能描述（目标、背景、期望行为、约束、验收线索等）。
