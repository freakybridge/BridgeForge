# 规划模式

## 定位与边界

只分析，不修改、创建或删除任何文件。计划经用户明确选择“开始实施”前，不得进入实现。

## 输入

以 `$ARGUMENTS` 为规划目标；为空时从当前用户请求提取目标和范围。

## 核心流程

1. 读取相关代码、配置、测试与文档，确认现状，不凭文件名猜实现。
2. 列出受影响的文件、类或函数；能定位时附行号。
3. 按 `P0 / P1 / P2` 编排编号任务，并为每项写明修改目标与验证方式。
4. 列出潜在风险、依赖、未知项和需要用户决定的分叉。
5. 给出完整计划后，使用当前平台的结构化提问工具结束本轮，选项固定为：
   - 开始实施
   - 调整计划
   - 放弃

## 输出与验证

输出必须包含：现状依据、影响文件、分步任务、验证方式、风险与停止点。不得把尚未运行的检查写成“已验证”。

## 停止条件

- 缺少会显著改变方案的关键输入：说明缺口并向用户确认。
- 用户未选择“开始实施”：停留在只读状态。

## 禁止事项

- 禁止在呈现计划的同一回合顺手实施。
- 禁止用泛泛建议代替对真实代码的检查。
- 禁止把未验证推断写成确定事实。
