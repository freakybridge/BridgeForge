# 分治协作模式

> 适用：涉及多文件、可拆成无依赖子任务并行推进的大型改动。
> **不适合**无法拆成无依赖子任务的改动；用户在 `confirm` 中选择 `collab` 后才进入本流程。

## 总原则：先确认需求，再拆分执行

研读（Explore 全量读码）+ 多 Agent 并行执行都是 token 大头。需求、范围、约束和验收统一由 `confirm` 在研读前确认；本 skill 只负责研读、拆分、最终确认和并行执行。

```
confirm → 研读与规划 → 拆分大纲与最终确认 → 并行执行 → 整合+独立验证
```

---

### Step 1: 确认卡硬闸

- 先检查本轮是否有 `confirm` 生成、且目标 / 范围 / 约束 / 验收均匹配的确认卡。
- 没有、任务不匹配或已过时：立即调用 `confirm`，在它返回确认卡前禁止研读、创建协作记录或 spawn agent。
- 有效卡存在：读取它，不得重复需求访谈，进入 Step 2。

### Step 2: 研读与规划（必要时补事实）

- 用 Explore agent 深入研读所有涉及的代码
- 研读中若发现事实缺口，只索取一项必要材料；若发现需求卡不匹配当前任务，返回 `confirm`，别带着错误假设拆分
- **创建协作记录文件**，放到对应子项目目录，命名格式 `collabs_YYYY-MM-DD_<简短主题>.md`：
  - **单 feature 任务**（项目内某个 feature 模块）→ 放到 `doc/1_plan/<feature>/collabs_YYYY-MM-DD_<topic>.md`，文件名内**不要再重复 feature 名**（子目录已暗示）
  - **跨 feature 任务** → 放到 `doc/1_plan/<topic>/multi-feature/`
  - **跨模块重构 / 底层任务** → 放到 `doc/1_plan/refactor/`
  - **其他主题** → 找匹配的子项目目录
  - 找不到合适子项目时新建子目录（如 `doc/1_plan/<topic>/<subproject>/`），并向用户说明新建原因
  - 子项目目录承载该项目的完整生命周期：design / collabs / acceptance 同位
- **研读后出口复查**：研读若确认无法拆成无依赖并行子任务，停止在规划阶段并报告原因；不得硬凑并行组或绕过最终确认直接改走单线。

### Step 3: 拆分方案大纲与最终确认

先给一个**简短拆分大纲**（不甩长文档）。

> ⛔ **硬契约**：把拆分大纲作为一个选择题交给用户确认或调整，本步以该题结束回合；用户未拍板前禁止 spawn 任何执行 Agent。

拆分大纲包含：
```markdown
## 协作计划

### 目标
{整体改动目标}

### 子任务拆分
| # | 子任务 | 负责文件 | 依赖 | 并行组 |
|---|--------|----------|------|--------|
| 1 | xxx    | a.py, b.py | 无  | A      |
| 2 | yyy    | c.py, d.py | 无  | A      |
| 3 | zzz    | e.py       | 1,2 | B      |

### 接口约定
{子任务之间的接口、数据结构、命名约定，确保各自独立开发后能正确衔接}
```

> **粒度由依赖驱动**：由 agent 按依赖、接口边界和单任务不超过 3-5 文件决定；这是执行细节，不重复向用户访谈。

### Step 4: 并行执行

按并行组依次执行：
- 同一并行组内的子任务**同时启动多个 Agent 并行执行**
- 每个 Agent 收到的 prompt 必须包含：
  1. 整体目标概述
  2. 该子任务的具体要求
  3. 接口约定（确保与其他子任务衔接）
  4. 明确的文件边界（只允许修改哪些文件）
- 前一并行组全部完成后，才启动下一组
- **失败重试上限**：某子任务失败，主 Agent 最多重试 1 次（共 2 次尝试）；仍失败则**停下来上报用户**，说明卡点，不得闷头循环重试（**比全局鬼打墙红线「同一 bug 前 3 次失败、第 4 次禁动手」（CLAUDE.md / AGENTS.md）更严**，方向一致）

### Step 5: 整合 + 独立验证

所有子任务完成后：
1. 主 Agent 处理衔接代码（import、注册、配置串联）并汇总到协作记录文件
2. **启动一个独立 review Agent** 复查整合正确性。它的 prompt 必须包含：① 整体目标 ② 接口约定 ③ 本次改动涉及的文件清单。并**硬性要求它独立 Read 实际改动 / 跑 `git diff` 自行核对**，检查接口衔接是否符合约定、有无遗漏的串联、是否引入回归 —— **不得只复核主 Agent 的文字结论**（那是自审回声），必须看真实代码。
3. 运行相关测试验证整体可用性
4. review Agent 的结论写入协作记录文件；若发现问题，回到 Step 5 修复（review 打回的修复**不计入**子任务「重试 1 次即停」配额，那是另一类问题）

## 协作记录格式

```markdown
# 协作：{主题}
> 日期：{YYYY-MM-DD}
> 目标：{一句话目标}
> 确认卡：{路径} / 范围 / 硬约束：{确认卡摘要}

## 协作计划
{拆分表格 + 接口约定}

## 执行记录

### 并行组 A
- **子任务 1**：{状态} — {简要说明改了什么}
- **子任务 2**：{状态} — {简要说明改了什么}

### 并行组 B
- **子任务 3**：{状态} — {简要说明改了什么}

## 整合 + 独立验证
- [ ] 接口衔接检查（主 Agent）
- [ ] import / 注册 / 配置串联
- [ ] 独立 review Agent 复查结论：{...}
- [ ] 测试通过

## 完成总结
{最终改动概览}
```

## 重要规则

1. **文件不重叠**：同一并行组内的 Agent 绝对不能修改同一个文件，这是硬性约束
2. **接口先行**：拆分时必须先定义好子任务间的接口约定（函数签名、数据结构、事件名等）
3. **确认卡硬闸**：没有有效 `confirm` 需求卡，禁止研读、创建记录或启动执行 Agent
4. **粒度由依赖驱动**：按依赖和接口边界切分，单子任务上限 3-5 文件
5. **失败重试上限 2 次**：超过即停下上报用户，不闷头循环
6. **独立验证**：整合后由独立 review Agent 复查，不靠主 Agent 自审
7. **全程记录**：执行过程实时更新到协作记录文件
8. **最终确认**：拆分大纲必须以一个选择题获得确认，才可启动执行 Agent
