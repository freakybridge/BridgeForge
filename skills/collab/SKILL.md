# 分治协作模式

> 适用：涉及多文件、可拆成无依赖子任务并行推进的大型改动。
> **不适合**单线就能做完的小改动 —— 那种直接做，别套重型流程（见 Step 1 轻量出口）。

## 总原则：渐进对齐，每道闸都停下等用户

研读（Explore 全量读码）+ 多 Agent 并行执行都是 token 大头。流程被切成几道「闸」，每道闸都卡在烧 token 的动作**之前**停下等你拍板，拆分方向一偏立即纠正，绝不整片白烧。

```
闸① 引导发问 → 闸② 复述确认 → 研读(可追问) → 闸③ 拆分大纲 → 并行执行 → 整合+独立验证
```

---

### Step 1: Welcome + 边界 + 引导发问（闸①，研读前）

先向用户**简述**：本模式会做什么、适用边界、大致会起多少 Agent / token 量级。然后用 **AskUserQuestion** 抛出引导性问题对齐目标：

1. **完成定义**：这次改动「做完」的验收标准是什么？（要支持什么、跑通什么）
2. **拆分粒度偏好**：倾向「少而大」（子任务少、各自范围宽）还是「多而细」（子任务多、各自小巧）？
3. **硬约束**：哪些文件 / 接口 / 行为绝对不准动？有无必须遵守的既有约定？

> ⛔ **硬契约**：本步必须以 **AskUserQuestion 调用结束当前回合**，用户未回答前**禁止**研读 / spawn 任何 agent / 往下走任何一步。

> **轻量出口**：若发现任务其实很小（粗判据：预计 ≤2 文件、单线即可完成）、或根本无法拆成无依赖的并行子任务，**直接建议「单线做就行，不必起分治」并退出**，不硬走完流程。

### Step 2: 复述确认（闸②）

用一两句话**复述**你理解的整体目标 / 范围 / 硬约束，作为 **AskUserQuestion 抛给用户**（「我理解的目标/范围是 X，对吗？」）。

> ⛔ **硬契约**：本步必须以 **AskUserQuestion 调用结束当前回合**，**不得**复述完就继续研读。用户点头前**禁止进入 Step 3**。闸①②虽相邻，也**绝不可合并**为一回合连跑。

### Step 3: 研读与规划（可追加发问）

- 用 Explore agent 深入研读所有涉及的代码
- 研读中若发现关键信息缺口，**立即用 AskUserQuestion 追问**，别带着错误假设拆分
- **创建协作记录文件**，放到对应子项目目录，命名格式 `collabs_YYYY-MM-DD_<简短主题>.md`：
  - **单 feature 任务**（项目内某个 feature 模块）→ 放到 `doc/1_plan/<feature>/collabs_YYYY-MM-DD_<topic>.md`，文件名内**不要再重复 feature 名**（子目录已暗示）
  - **跨 feature 任务** → 放到 `doc/1_plan/<topic>/multi-feature/`
  - **跨模块重构 / 底层任务** → 放到 `doc/1_plan/refactor/`
  - **其他主题** → 找匹配的子项目目录
  - 找不到合适子项目时新建子目录（如 `doc/1_plan/<topic>/<subproject>/`），并向用户说明新建原因
  - 子项目目录承载该项目的完整生命周期：design / collabs / acceptance 同位
- **研读后出口复查**：研读常常才暴露「其实无法拆成无依赖并行子任务」。**若此刻确认不可并行拆，立即退出建议单线做**，不要硬凑出并行组（闸① 的出口只能粗判，这里才是最准的判断点）。

### Step 4: 拆分方案大纲（闸③）

先给一个**简短拆分大纲**（不甩长文档）。

> ⛔ **硬契约**：把拆分大纲作为 **AskUserQuestion 抛给用户确认或调整**，本步以该工具调用**结束回合**；用户未拍板前**禁止 spawn 任何执行 Agent**。

拆分大纲包含：
```markdown
## 协作计划

### 目标
{整体改动目标}

### 子任务拆分
| # | 子任务 | 负责文件 | 依赖 | 并行组 |
|---|--------|----------|------|--------|
| 1 | xxx    | a.py, b.py | 无  | A      |
| 2 | yyy    | c.py, d.py | 无  | A      |
| 3 | zzz    | e.py       | 1,2 | B      |

### 接口约定
{子任务之间的接口、数据结构、命名约定，确保各自独立开发后能正确衔接}
```

> **粒度由闸① 偏好驱动**：按用户选的「少而大 / 多而细」决定切多细；无论如何单个子任务以不超过 3-5 个文件为上限，再大就继续拆。

### Step 5: 并行执行

按并行组依次执行：
- 同一并行组内的子任务**同时启动多个 Agent 并行执行**
- 每个 Agent 收到的 prompt 必须包含：
  1. 整体目标概述
  2. 该子任务的具体要求
  3. 接口约定（确保与其他子任务衔接）
  4. 明确的文件边界（只允许修改哪些文件）
- 前一并行组全部完成后，才启动下一组
- **失败重试上限**：某子任务失败，主 Agent 最多重试 1 次（共 2 次尝试）；仍失败则**停下来上报用户**，说明卡点，不得闷头循环重试（**比全局鬼打墙红线「同一 bug 前 3 次失败、第 4 次禁动手」（CLAUDE.md §8）更严**，方向一致）

### Step 6: 整合 + 独立验证

所有子任务完成后：
1. 主 Agent 处理衔接代码（import、注册、配置串联）并汇总到协作记录文件
2. **启动一个独立 review Agent** 复查整合正确性。它的 prompt 必须包含：① 整体目标 ② 接口约定 ③ 本次改动涉及的文件清单。并**硬性要求它独立 Read 实际改动 / 跑 `git diff` 自行核对**，检查接口衔接是否符合约定、有无遗漏的串联、是否引入回归 —— **不得只复核主 Agent 的文字结论**（那是自审回声），必须看真实代码。
3. 运行相关测试验证整体可用性
4. review Agent 的结论写入协作记录文件；若发现问题，回到 Step 5 修复（review 打回的修复**不计入**子任务「重试 1 次即停」配额，那是另一类问题）

## 协作记录格式

```markdown
# 协作：{主题}
> 日期：{YYYY-MM-DD}
> 目标：{一句话目标}
> 范围 / 硬约束：{闸①② 对齐结果}

## 协作计划
{拆分表格 + 接口约定}

## 执行记录

### 并行组 A
- **子任务 1**：{状态} — {简要说明改了什么}
- **子任务 2**：{状态} — {简要说明改了什么}

### 并行组 B
- **子任务 3**：{状态} — {简要说明改了什么}

## 整合 + 独立验证
- [ ] 接口衔接检查（主 Agent）
- [ ] import / 注册 / 配置串联
- [ ] 独立 review Agent 复查结论：{...}
- [ ] 测试通过

## 完成总结
{最终改动概览}
```

## 重要规则

1. **文件不重叠**：同一并行组内的 Agent 绝对不能修改同一个文件，这是硬性约束
2. **接口先行**：拆分时必须先定义好子任务间的接口约定（函数签名、数据结构、事件名等）
3. **三道闸（引导发问 / 复述确认 / 拆分大纲）必须各自停下等用户**，不得连贯跑完
4. **粒度由偏好驱动**：按闸① 的「少而大 / 多而细」决定切分，单子任务上限 3-5 文件
5. **失败重试上限 2 次**：超过即停下上报用户，不闷头循环
6. **独立验证**：整合后由独立 review Agent 复查，不靠主 Agent 自审
7. **全程记录**：执行过程实时更新到协作记录文件
8. **轻量出口**：对齐 / 复述阶段若发现任务太小或无法并行拆分，提前退出并建议单线做
