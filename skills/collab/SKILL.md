# 分治协作模式

## 流程

### Step 1: 研读与规划
- 用 Explore agent 深入研读所有涉及的代码
- **创建协作记录文件**：放到对应子项目目录，命名格式 `collabs_YYYY-MM-DD_<简短主题>.md`
  - **单 feature 任务**（项目内某个 feature 模块）→ 放到 `doc/1_plan/<feature>/collabs_YYYY-MM-DD_<topic>.md`，文件名内**不要再重复 feature 名**（子目录已暗示）
  - **跨 feature 任务** → 放到 `doc/1_plan/<topic>/multi-feature/`
  - **跨模块重构 / 底层任务** → 放到 `doc/1_plan/refactor/`
  - **其他主题**：`infra` / `performance` / `ui` / `meta` / `risk` 等下找匹配的子项目目录
  - 找不到合适子项目时新建子目录（如 `doc/1_plan/<topic>/<subproject>/`），并向用户说明新建原因
  - 子项目目录承载该项目的完整生命周期：design / collabs / acceptance 同位
- 制定拆分计划，写入记录文件

拆分计划必须包含：
```markdown
## 协作计划

### 目标
{整体改动目标}

### 子任务拆分
| # | 子任务 | 负责文件 | 依赖 | 并行组 |
|---|--------|----------|------|--------|
| 1 | xxx    | a.py, b.py | 无  | A      |
| 2 | yyy    | c.py, d.py | 无  | A      |
| 3 | zzz    | e.py       | 1,2 | B      |

### 接口约定
{子任务之间的接口、数据结构、命名约定，确保各自独立开发后能正确衔接}
```

### Step 2: 用户确认
将拆分计划展示给用户，等待确认或调整。**未经确认不得开始执行。**

### Step 3: 并行执行
按并行组依次执行：
- 同一并行组内的子任务**同时启动多个 Agent 并行执行**
- 每个 Agent 收到的 prompt 必须包含：
  1. 整体目标概述
  2. 该子任务的具体要求
  3. 接口约定（确保与其他子任务衔接）
  4. 明确的文件边界（只允许修改哪些文件）
- 前一并行组全部完成后，才启动下一组

### Step 4: 整合审查
所有子任务完成后，主 Agent 负责：
1. 审查各子任务的修改是否符合接口约定
2. 处理衔接代码（如 import、注册、配置串联）
3. 运行相关测试验证整体可用性
4. 将最终结果汇总到协作记录文件

## 协作记录格式

```markdown
# 协作：{主题}
> 日期：{YYYY-MM-DD}
> 目标：{一句话目标}

## 协作计划
{拆分表格 + 接口约定}

## 执行记录

### 并行组 A
- **子任务 1**：{状态} — {简要说明改了什么}
- **子任务 2**：{状态} — {简要说明改了什么}

### 并行组 B
- **子任务 3**：{状态} — {简要说明改了什么}

## 整合审查
- [ ] 接口衔接检查
- [ ] import / 注册 / 配置串联
- [ ] 测试通过

## 完成总结
{最终改动概览}
```

## 重要规则

1. **文件不重叠**：同一并行组内的 Agent 绝对不能修改同一个文件，这是硬性约束
2. **接口先行**：拆分时必须先定义好子任务间的接口约定（函数签名、数据结构、事件名等）
3. **用户确认**：计划必须经用户确认才能执行，不可自行开始
4. **粒度适中**：每个子任务不超过 3-5 个文件，太大则继续拆分
5. **失败隔离**：某个子任务失败不影响其他子任务，主 Agent 负责重试或调整
6. **全程记录**：执行过程实时更新到协作记录文件
