# confirm — 统一需求确认

定位：这是所有高级流程共用的“开工确认单”。先核对真实事实，再每轮只确认一个开发要点；只有用户确认完整卡片，才能写入 `doc/1_plan/<topic>/` 并交接后续工作。

## 调用方式与硬闸

- 用户可直接调用 `/confirm` 或 `$confirm`，用于理清需求。
- `develop`、`debate`、`collab` 启动时必须先检查本轮是否已有有效确认卡；没有、任务不匹配或已过时，就调用本 skill，禁止继续各自流程。
- 有效确认卡必须同时匹配当前任务的目标、范围、约束和验收；不得只因目录或主题名称相同就复用旧卡。
- 被高级 skill 调用时，确认卡落盘后直接返回调用方；用户直接调用时，落盘后用一题选择自动交接给 `develop`、`debate` 或 `collab`。

## Step 1：事实核验（先读，禁止确认）

当用户点名已有文件、表格、页面、报表、接口、数据库、业务流程或历史记录，且需求涉及重构、接入、扩展或替换时：
- 先只读核验点名载体的真实结构和直接关联的读写 / 输出入口；不能只依据旧代码、注释或猜测讨论需求。
- 表格 / 工作簿至少核验 sheet、表头、关键日期列、公式或格式，以及有代表性的历史行；页面 / 报表 / 接口至少核验当前字段、触发入口和输出语义。
- 先输出“已核实事实 / 未核实事实”。本阶段禁止输出确认卡、写 `doc/` 或改代码。
- 载体不可访问或权限不足时，明确报告卡点；用户明确要求纯概念讨论时，可继续，但未读载体相关结论必须标为假设。

## Step 2：单题选择式需求确认

先从已核实事实中列出实际涉及且会影响实现或验收的开发要点。典型要点包括目标与范围、主记录单位、用户动作与时点、字段 / 数据映射、状态与例外、历史保留 / 覆盖语义、非目标、验收标准；不相关的要点不问。

每轮只能确认一个要点，且必须使用以下选择题：

```text
问题：{只确认一个开发要点的具体问题}
A. {基于已核实事实的选项}
B. {互斥选项}
C. {互斥选项；没有第三种合理路径时可省略}
D. 其他：{用户补充真实需求}
```

- 一条消息只允许有这一个问题；禁止附带第二题、问题清单、确认卡或“顺便确认”。
- 选项必须具体、互斥且能改变后续实现；不得把多个独立要点塞进同一选项，也不得用抽象技术方案替代业务确认。
- 对既有载体，选项必须锚定已核验字段、样例或流程节点，优先逐项确认记录单位、填写时点和角色、状态 / 例外、历史保留与覆盖语义。
- 用户选择或补充后，记录结论，再只提出下一个未确认要点。用户回答“其他”时，先整理为当前要点的候选选项，仍以单题选择确认，不擅自解释为最终需求。
- 用户回答“暂不确定”时，单题确认它是阻塞开发、采用临时默认，还是移出本轮范围。影响用户行为、数据语义、接口或验收的临时默认，必须再单题确认具体默认规则。
- 不设问答数量上限。只有所有实际涉及要点都已确认、明确待定处理或被用户排除，才能进入 Step 3。

## Step 3：完整确认卡与落盘

先展示以下完整确认卡：

```text
目标：这次要解决什么
不做：这次明确不碰什么
已核实事实：载体路径、结构、现有流程的关键事实
已确认业务规则：记录单位、填写时点、状态 / 例外、历史语义
数据映射：输入 / 输出与既有载体字段的映射，或明确待新增字段
拟修改：预计会新增 / 修改哪些文档、代码、配置类别
验收：做完怎么判断对
合理假设与剩余风险：哪些点未验证，不能混入已确认项
自动化边界：确认后会写入需求卡并交接指定高级 skill；中途只在卡点 / 高风险 / 完成时打断
```

然后只问一个选择题：

```text
问题：以上是否准确表达你的真实开发需求？
A. 准确，写入需求卡并继续
B. 有一项不准确，继续逐项确认
C. 暂不写入，只保留讨论结论
```

- 仅当完整确认卡已展示，且用户针对该题明确选择 A 或作出唯一对应 A 的无条件同义回答时，才可写入。
- 用户选 B 时，必须回到一个具体开发要点；带新增条件、保留意见、多个选项或无法唯一映射的回答一律不算确认。
- 用户选 A 后，写入 `doc/1_plan/<topic>/requirements_YYYY-MM-DD_<slug>.md`，状态为 `confirmed`；新建文档必须同步 `doc/README.md`。
- 需求卡至少包含本节八项、原始需求摘要、调用来源、后续交接目标、实施记录和验证记录占位。

## Step 4：交接

- 被 `develop`、`debate` 或 `collab` 调用：返回调用方和已确认需求卡路径；调用方只能读取该卡并进入自身专业流程。
- 用户直接调用：写入后只问一个选择题：

```text
问题：将这份已确认需求交给哪个流程继续？
A. develop：落盘、实现、验证与试用闭环
B. debate：研读、双 Agent 辩论并形成方案
C. collab：规划无依赖子任务并并行执行
```

用户选择后自动交接；不得重复需求访谈。

## 红线

1. 没有有效确认卡的高级 skill 必须转入 `confirm`，不得自行补问几句后继续。
2. 每轮只问一个开发要点，必须是选择题；不得因“看起来简单”跳过确认。
3. 未得到 Step 3 的最终确认前，禁止写需求卡、改代码或启动高成本 Agent。
4. 确认卡不匹配当前任务或已过时，视同没有确认卡，必须重新确认。
5. 不自动 `git add` / `commit` / `push`，除非用户明确要求。
