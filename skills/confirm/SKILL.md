# confirm — 统一需求确认

## 定位与边界

作为高级开发流程的开工确认单：先核验真实载体，再逐项确认需求。用户最终确认完整卡片之前，禁止写需求文档、改代码或启动高成本 agent。

## 输入

- 用户的原始目标、点名载体、已知约束和验收线索。
- 本轮已有确认卡（如有），用于判断目标、范围、约束和验收是否仍匹配。
- 被 `develop`、`debate` 或 `collab` 调用时的交接目标。

## 核心流程

### 1. 核验事实

1. 用户点名文件、表格、页面、报表、接口、数据库、业务流程或历史记录时，先只读真实载体及直接关联的读写 / 输出入口。
2. 表格至少核验 sheet、表头、关键日期列、公式或格式和代表性历史行；页面、报表与接口至少核验当前字段、触发入口和输出语义。
3. 输出“已核实事实 / 未核实事实”。此阶段禁止展示确认卡或实施方案。
4. 载体不可访问时报告卡点；用户明确要求概念讨论时可以继续，但相关结论必须标为假设。

### 2. 逐项确认

1. 从真实事实中识别会改变实现或验收的要点，例如目标与范围、记录单位、用户动作与时点、字段映射、状态与例外、历史语义、非目标和验收标准。
2. 每轮只问一个要点，并使用具体、互斥、会改变实现的选择题：

```text
问题：{一个具体开发要点}
A. {基于已核实事实的选项}
B. {互斥选项}
C. {可选的第三条合理路径}
D. 其他：{用户补充真实需求}
```

3. 用户选“其他”时，把补充整理为当前要点的新选项，再单题确认；不得直接当作最终结论。
4. 用户暂不确定时，单题确认该点是阻塞、采用临时默认还是移出范围；影响用户行为、数据、接口或验收的默认规则必须再次确认。
5. 所有实际要点都确认、明确待定处理或被排除后，才进入完整确认卡。

### 3. 确认并落盘

1. 展示包含以下内容的完整确认卡：目标、不做、已核实事实、已确认业务规则、数据映射、拟修改、验收、合理假设与风险、自动化边界。
2. 只问以下一题，并结束当前回合：

```text
问题：以上是否准确表达你的真实开发需求？
A. 准确，写入需求卡并继续
B. 有一项不准确，继续逐项确认
C. 暂不写入，只保留讨论结论
```

3. 只有用户明确选择 A 或给出唯一、无条件对应 A 的回答，才写入 `doc/1_plan/<topic>/requirements_YYYY-MM-DD_<slug>.md`，状态设为 `confirmed`；同步 `doc/README.md`。
4. 需求卡必须包含上述确认内容、原始需求摘要、调用来源、后续交接目标以及实施 / 验证记录占位。

### 4. 交接

- 被高级 skill 调用时，返回确认卡路径，由调用方继续，不重复访谈。
- 用户直接调用时，落盘后只问一题选择 `develop`、`debate` 或 `collab`；用户选择后交接。

## 输出与收据

- 事实核验阶段：已核实与未核实事实。
- 落盘阶段：确认卡路径、状态和 `doc/README.md` 索引更新。
- 交接阶段：目标高级 skill 和确认卡路径。

## 停止条件

- 每个选择题发出后立即结束当前回合，等待用户回答。
- 点名载体不可访问且事实会影响需求时，报告卡点并停止确认该项。
- 确认卡任务不匹配或已过时，视同无卡，重新进入本流程。

## 禁止事项

- 禁止一轮提出多个开发问题或附带问题清单。
- 禁止以抽象技术方案代替业务规则确认。
- 禁止最终确认前写需求卡、改代码或启动高成本 agent。
- 禁止高级 skill 绕过本流程自行补问后继续。
- 禁止自动 `git add`、commit 或 push。
