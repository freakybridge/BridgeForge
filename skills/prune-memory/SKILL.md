# /prune-memory — 清理 MEMORY.md

## 目标

把 MEMORY.md 的总行数降到 185 行以内，同时保留所有仍有召回价值的条目。

## 删留标准

**删除（满足任一）**：
1. **已升入 rules**：章节标注"已升级为 path-rule"的条目，rules 是单一事实源，memory 是重复索引
2. **一次性历史事件**：特定 bug 已修、根因已消失，不会以同样形式复现
3. **代码即文档**：约束已硬编码进代码/配置，不需要额外记忆
4. **已完结阶段**：描述某个已完成且不会回头的迁移/过渡方案

**保留（满足任一）**：
1. **非显而易见的约束**：未来不读这条就会重犯
2. **可被推翻的架构决策**：不记录就会被重新讨论
3. **跨 Gateway/模块的通用规律**：适用范围广、召回价值高

**速判方法**：问"如果没有这条，下次会怎样？"
- 会踩同一个坑 → **留**
- 会去看 rules 文件 → **删**（rules 已覆盖）
- 什么都不会发生 → **删**

## 执行流程

### Step 1：读取现状
读 `MEMORY.md`，报告当前行数和各章节条目数。

### Step 2：逐章节分析
对每个章节，按标准分类每一条：
- 🔴 建议删（说明理由）
- 🟡 不确定（说明疑虑，让用户决定）
- 🟢 保留

优先处理行数最多的章节。

### Step 3：展示候选清单

格式（每条一行，紧凑）：
```
🔴 feedback_nanhua_frontend_ip_migration_2026 — 南华 IP 搬迁一次性事件，根因已消失
🔴 project_ctp_initial_sync_baseline — 基准数字已在代码注释，无召回价值
🟡 project_ctp_multi_sdk_version — 标注"待实现"，是否仍在计划中？
```

### Step 4：等用户确认

用户可以：
- 逐条回复 y/n
- 说"红色全删"批量批准
- 说"跳过不确定"只处理红色

### Step 5：执行删除

1. 从 MEMORY.md 删除对应索引行（Edit 工具）
2. 对应子文件：移到 `memory/archive/` 子目录（`git mv`），不直接删除

### Step 6：验证结果

统计删除后行数。如果仍 > 185，继续分析下一批候选。

## 红线

- ❌ 禁止自动删除——必须用户逐条或批量确认
- ❌ 禁止直接删除子文件——移到 `archive/` 保留 trail
- ❌ 删完必须验证行数，没达标继续清理
