# 跨项目写入事故与 hook 防护提案

状态：pending  
日期：2026-07-10  
来源：ClaudeBridgeAssist 对话框误修改 StratusAgent 代码后的事故复盘

## 结论

BridgeForge 应在 Claude 与 Codex 两套骨架中默认加入跨项目写入保护：当当前对话框归属项目为 A，但工具即将写入、删除、执行 git 操作或修改 B 项目时，PreToolUse hook 必须阻断，并要求用户明确确认目标项目。

这类事故的关键不是文件系统权限，而是用户心智模型。用户看到的是“A 项目的对话框”，但 agent 可能在历史上下文、绝对路径、外部授权或跨目录命令的作用下改到 B 项目。hook 应把这种“跨项目边界”在动手前显式暴露出来。

## 事故背景

在 ClaudeBridgeAssist 对话框中，用户连续验收 StratusAgent 的手动账户功能。agent 根据用户的“开始改吧 / 开始吧”直接修改了 StratusAgent 代码。之后用户才发现当前对话框其实属于 ClaudeBridgeAssist 项目。

从结果看，代码修改可能是用户当时想要的；但从流程看，这是事故：

- 对话框归属项目与实际被修改项目不一致。
- 跨项目修改前没有醒目确认。
- 用户直到事后才意识到 B 项目被改过。
- 现有权限模型只控制“能不能写”，没有控制“该不该在这个项目语境里写”。

## 风险定义

高风险操作包括：

- 写入当前项目根目录之外的文件。
- 删除、移动、重命名当前项目根目录之外的文件。
- 在当前项目之外执行 `git add`、`git commit`、`git restore`、`git reset`、`git push` 等 git 操作。
- 在当前项目之外运行格式化、测试、构建、迁移脚本，且这些命令可能产生文件改动。
- 使用绝对路径把报告、规则、memory、代码写入另一个项目。

这些操作即使经过 sandbox escalation，也仍然需要项目语义层面的确认。

## 目标

1. Claude 骨架默认启用跨项目写入保护。
2. Codex 骨架默认启用跨项目写入保护。
3. 当前项目 A 的对话框默认不得静默修改项目 B。
4. 读外部项目可以弱提示或记录；写外部项目必须强确认。
5. 用户确认时必须看到当前项目、目标项目和操作类型。

## 建议实现

### 1. 会话启动时记录项目根

SessionStart hook 或等价初始化逻辑记录：

- 当前工作目录。
- 当前 git 仓库根目录。
- 项目显示名。
- 可选：项目唯一标识，例如 `.git` root、repo remote、workspace id。

后续所有工具调用都以这个项目根作为“当前对话框归属项目”。

### 2. PreToolUse 检查目标路径

对 shell、文件写入、patch、git、脚本执行类工具做路径分析：

- 如果所有写入目标都在当前项目根内：放行。
- 如果只是读取外部路径：允许，但建议记录一次外部读取事件。
- 如果存在当前项目根外的写入目标：阻断。
- 如果命令在外部项目根执行 git 操作：阻断。

阻断提示必须包含：

- 当前项目：例如 `D:\Quant\ClaudeBridgeAssist`
- 目标路径或目标项目：例如 `D:\Quant\StratusAgent`
- 操作类型：写文件、删文件、git 操作、构建可能写入等
- 要求用户明确确认：“当前对话属于 A，但将修改 B。是否继续？”

### 3. 明确确认后才允许一次性放行

建议确认语义是一次性的，不要永久放行。

可接受：

- 本次工具调用放行。
- 本轮任务内对同一目标项目放行，但需要在状态中记录。

不建议：

- 对某个绝对路径永久放行。
- 只因为用户授权了 filesystem escalation 就跳过项目边界确认。

### 4. 两套骨架同步落地

需要覆盖：

- `templates/claude/...` 的 hook 配置与实现。
- `templates/codex/...` 的 hook 配置与实现。
- 两套骨架的 `CLAUDE.md` / `AGENTS.md` 或 rules 中加入明确红线。
- BridgeForge 安装、切换、更新流程中确保该 hook 不丢失。

建议共享核心检测脚本，由 Claude 与 Codex 两侧薄封装调用，避免两套骨架长期漂移。

## 建议的分级策略

### 低风险：外部只读

例如读取上游文档、查看另一个项目的 git status。

建议：允许，但在 agent 输出中提示“正在只读访问外部项目”。

### 中风险：外部写文档

例如从 A 项目对话框给 B 项目写报告。

建议：阻断并要求确认。确认文案必须说清“这是跨项目写入”。

### 高风险：外部改代码

例如从 A 项目对话框修改 B 项目源码、配置、测试。

建议：强阻断。用户确认后仍需 agent 在动手前说清目标项目。

### 极高风险：外部 git 操作或破坏性操作

例如在 B 项目执行 commit、push、restore、reset、删除目录。

建议：强阻断，并叠加现有高风险操作确认规则。

## 豁免建议

可豁免或弱化的情况：

- 当前工作区本身就是多仓 monorepo，且目标路径仍在声明的 workspace root 下。
- 用户显式说“到 B 项目写报告 / 修改 B 项目”，但仍建议首次跨项目写入时提示一次。
- 临时目录、系统缓存目录、构建输出目录，但必须有明确用途。
- 只读命令。

不应豁免：

- 因为命令用了绝对路径。
- 因为 sandbox escalation 已获批。
- 因为历史上下文里用户曾经提过另一个项目。
- 因为 agent 自认为“这就是用户要的”。

## 测试建议

至少覆盖：

- 当前项目内写文件：放行。
- 当前项目外写报告：阻断并显示当前项目与目标项目。
- 当前项目外改源码：阻断。
- 当前项目外 `git status`：放行或弱提示。
- 当前项目外 `git commit`：阻断。
- 用户明确确认后，同一次操作可放行。
- Claude 模板安装后 hook 生效。
- Codex 模板安装后 hook 生效。
- BridgeForge switch 后 hook 仍保留。

## 严重性

建议按 P0/P1 之间处理，偏 P0。

理由：跨项目静默写入会直接破坏用户对“这个对话框属于哪个项目”的基本信任。即使改动内容正确，流程上也是越界；一旦改错项目，后果可能包括错误提交、错误发布、污染上游模板或污染下游业务代码。

## 与非 ASCII 传输保护的关系

这两个问题都适合用 hook 解决，但保护对象不同：

- 非 ASCII 传输保护：防止文本在 shell 中转链路中损坏。
- 跨项目写入保护：防止 agent 在错误项目语境中修改外部项目。

两者可以共享同一类实现原则：

- PreToolUse 先拦截高风险动作。
- 错误发生前 fail-fast。
- 不依赖 agent 自觉。
- Claude 与 Codex 两套骨架必须同时具备保护。
