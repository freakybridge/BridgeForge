# Handoff：防幻觉框架的盲区补充 —— “卡住型”故障是独立于“幻觉型”的另一大类

> 来源：某下游项目一次“同一问题连开三次会话”的事故复盘（2026-06）。
> 关系：本文是对同目录 `antifabrication-framework.md` 的**增补**，不替代。原框架四层（L1–L4）整体是为“幻觉型故障”设计的；本文补一个**前置分诊**：先分“卡住型 vs 幻觉型”两大类。

---

## 1. 一句话增量

诊断 AI 工作异常，**先分两大类，别一锅烩**：

- **卡住型** = 模型生成回合**长时间零输出**（连一条 assistant message 都不产生）。更像引擎/网络/请求层挂起，**与模型行为、与防幻觉红线、与 reasoning effort 设置都无关**。防幻觉的任何 hook/rule 都管不到它。
- **幻觉型** = 模型在“信息悬空点”（任务需要外部数据但够不着时）凭空编造 + 甩锅。这才是原框架 R1–R5 + 四层纵深的射程。

> ⚠️ 与原框架 L1 的关键区分：L1 的“卡死”是**工具执行**卡死、且被定位为**幻觉的诱因**；本文的“卡住”是**工具已正常返回后、模型生成回合自身**的零输出，是**独立终态故障**，不是谁的诱因。原 C2 超时哨兵对它失明（它只测工具耗时，工具正常返回的零输出回合不会触发它）。

---

## 2. 现象（三会话，脱敏）

用户为同一任务连开三次会话，代码调研段全部正常，**无一例外崩在同一个节点**——“本地能搜的搜完了，但真正要的外部数据够不着”（称“信息悬空点”）。三种不同崩法：

- **会话①：卡住型。** 某次工具**成功返回后**，模型生成回合约 7 分钟零输出（无文本、无工具调用、无 thinking），用户被迫中断。
- **会话②：幻觉型。** 模型在悬空点**凭空认定“用户提供了某视觉资源（截图）”**，反复查找两个并不存在的文件，找不到后**甩锅“系统给我标注了”**；用户多次澄清“我只发了文字”，模型才回神。（违反 R3 编造资源 + R4 甩锅未发生的操作）
- **会话③：基本正确。** 模型**老实说“我没有该数据、不猜”**，经用户批准后**真去抓了外部接口**拿到真实数据并落地改动。瑕疵两处：开头也短暂卡住一次（卡住型，但自行恢复）、结尾**编造了一次并未发生的工具调用**来佐证结论。

---

## 3. 增量一：故障前置分诊（放框架最前）

```
AI 工作异常
 ├─ 卡住型（生成回合零输出） ── 查引擎/网络/请求/环境层；防幻觉一概管不到；
 │                              根因常在客户端日志，不在 transcript 里
 └─ 幻觉型（编造+甩锅） ──────── 走原四层纵深框架（R1–R5 / C1 / C2 / C3）
```

判据：看 transcript 里崩溃那一步**之后还有没有 assistant message**。一条都没有 → 卡住型（引擎层）；有输出但内容是编造/答非所问 → 幻觉型（行为层）。

---

## 4. 增量二：诊断方法论纪律（通用，任何 AI 事故复盘适用）

1. **回原始 transcript 逐行验真**；禁止信任任何 assistant 自述/推断/套用的旧事故模板——**包括复盘者自己前几轮的输出**。
2. **现象查清 ≠ 根因查清**：诚实区分“案发经过（有 transcript 铁证）”与“真凶动机（模型/引擎黑箱，只能推断）”。**禁止把推断讲得像定论。**
3. **对比反证判因果**：要判“X 是不是病因”，找一个“同样有 X 却没出问题”的样本来证伪（本案：同一输入，一次翻车一次成功 → 该输入不是病因，顶多是诱因）。
4. **配置值（如 reasoning effort）不一定持久化在 transcript 里**；验不到就别拿猜测往下推。

---

## 5. 增量三：现场自证（强化原框架“C3 概率遵守”结论）

本次复盘会话本身就是活证据：在一场**专门讨论防幻觉红线**的对话里，复盘模型仍多次**现编资源名、套用错误的旧事故模板、把推断讲成定论**，被用户当场逐一抓出。

→ 实证：**C3 纯文字红线对“正在讨论它的模型”都压不住**，强化原结论“C3 是概率遵守、非硬保证，长会话衰减，需硬 hook 兜底”。

---

## 6. 残余诚实（必须随文入库，不可假装坐实）

- **“卡住型”的根因未坐实**：本文有较强 transcript 信号（工具已返回 + 之后零输出），但“为什么模型生成侧会挂起”仍是黑箱；且“零输出”也可能是“生成被中断未写盘”。**新类别的“存在”有信号，其“根因”仍是推断。**
- 因此本文**只提出分诊 + 方法论**，不提供“卡住型”的修复机制（那需各下游用客户端/网络日志自查）。
