2026-07-01 起，用户提出「先访谈式一问一答摸清 harness 偏好，再结合外部资料打造 harness 工程」的新方向（此前从未系统讨论）。承接 [[harness-trim-2026-07-01-deferred]]。本文件逐条追加访谈结论。

## 偏好访谈记录

**Q1 · 首要目标**：这套优化后的 harness 最想先变好在哪？
→ **防幻觉/谎报(B类) + 防空转，两个都要**（其余"瘦身减噪 / 贴合工作节奏"排后，非不做）。这正是最初 AB 对话调查报告点名的两大毛病（空转 + 幻觉）。

**Q2 · 闸的硬度**：逻到我快谎报/空转时，要硬拦还是软提醒？
→ **硬闸优先**（能拦死就拦死，偶尔误伤/打断可接受）。
- **重要含义**：这直接**松动** `antifabrication-framework.md` 里"C1 deny / C2 哨兵 / Stop 甩锅自检三 hook 因'误伤是硬停'而不进产品层"的老决策——用户既接受误伤，那三个被否的 hook 值得重新评估。
- **技术现实待消化**：B类谎报**能**硬 hook（PostToolUse/Stop 拿真实终态对账）；但**空转**基本**无法**硬 hook（模型 thinking 期无工具边界可挂、被 SUSPENDED），需单独想打法。见 [[feedback-llm-suspended-during-tool-exec]]。

**Q3 · 服务范围**：只调自身，还是下沉所有下游？
→ **产品层同步**（一开始就进 templates 下沉全下游）。
- **张力**：Q2"硬闸优先+误伤可接受" × Q3"下沉全下游" = 硬闸会**误伤每个下游**，正是 framework 当年否 C1/C2/Stop 的核心理由（"复印全下游 + 误启用风险"）。
- **可能解**：用**确定性对账**式硬闸（查"文件真写没/命令 exit/git 终态"这类事实，非 LLM 判断）→ 既硬又近零误伤 → 下沉安全。这是 trim-workflow 合成里指的方向，待 Q4 确认。

**Q4 · 误伤底线**：下沉的防谎报硬闸该多谨慎？
→ **只认铁证**（只查"文件真写没/exit 码/git 终态"这类硬事实，几乎不误伤）。**Q2×Q3 张力就此解开**。
- **设计骨架已定**：B类谎报 = **确定性对账硬闸**（PostToolUse/Stop 拿真实终态 vs 我声称做的动作比对，不一致就拦）。代价：只拦"留下事实痕迹的谎"（如谎称已改文件/已同步 memory）；纯嘴上的谎（编个不存在的 API 名当结论）仍无铁证可查，得靠别的层。
- **空转仍悬**：空转是内部 thinking，没有"铁证"可查（无工具边界、无文件痕迹）→ 在"只认铁证"原则下无法硬 hook，待 Q5 定打法。

**Q5 · 空转打法**：空转没铁证、硬闸拦不住，怎么打？
→ **找代理信号做软提醒**（用耗时 / token 增量当间接证据，提醒"是不是空烧了"）。**非硬闸**（空转天然只能软）。
- 现成先例可复用：`context_warning.py` 已在读 transcript 的真实 usage → 同款机制可扩一个"长 thinking 无产出"的代理探针。

**防护地图（两半齐）**：幻觉(B类) = 确定性对账**硬闸**；空转 = 代理信号**软提醒**。

**Q6 · 工程量预算**：新防护可能加几个 hook，预算多少？
→ **少而精 + 复用现有**（尽量挂已有 hook / pre-commit，少新建，维护负担第一）。
- 与 v0.38.0 pre-commit 花名册闸同路；也印证 trim-workflow 批评的"别新建重 ledger hook、复用 pre-commit + `git diff --cached`"。落地首选：**扩现有 `.githooks/pre-commit`**，而非造 PostToolUse ledger。

**Q7 · 硬闸首个靶子**：先对账哪种谎报？
→ **三个都要**（谎称已同步 memory/rules/doc、谎称已改/删文件、谎称测试/命令跑通）。
- **设计注记**：前两个（已同步 / 已改删）天然能靠 commit 时 `git diff --cached` 一网打尽（共用底座，合"少而精"）；第三个（测试跑通）得单独记命令 **exit 码**（PostToolUse on Bash，不同机制）→ 与"少而精"略张力，设计时尽量共用底座。

## 设计骨架（7 问汇总，待合成外部资料成正式方案）
- **靶**：幻觉(B类谎报) + 空转，两个都攻。
- **幻觉**：确定性对账**硬闸**，只认铁证（git/文件/exit），近零误伤 → 安全下沉全下游。对账三类谎报。主体复用 `.githooks/pre-commit` + `git diff --cached`。
- **空转**：无铁证 → 代理信号**软提醒**（耗时/token 增量，复用 context_warning 机制）。
- **工程**：少而精、复用现有，别造新重 hook。
- **下沉**：产品层 templates/，dogfood 镜像自身 .claude/。

## 关键转向（2026-07-01，Q7 后）
用户意识到「harness 工程不止防幻觉/防空转，需要一个**更系统的 harness 设计**」——防幻觉/防空转只是两个症状（对应下方九维地图的第 3+4 维）。方向从"补两个洞"升级为"先画全 harness 设计空间的地图 → 看现有覆盖/缺口/重叠 → 按维度系统性设计"。上面 7 问的骨架降级为"其中一个维度的偏好"，不是全部。

## 系统性访谈 · 从头开始（2026-07-01）
用户要求「从头开始、一个个问、逐一细节确认、全程白话解释」。我给的九维地图（① 澄清&范围 ② 方法纪律 ③ 失败处置[含空转] ④ 交付真实性[含防幻觉] ⑤ 跨会话状态&知识 ⑥ 知识治理 ⑦ 协作模式 ⑧ 卫生&可移植 ⑨ 人机沟通 + 纵轴力度 hook>rule>CLAUDE.md）作为初步框架。防幻觉/防空转 = 第 3+4 维，只是一部分。

**N1 · 北极星**：养这套 harness 最想换来的第一位？
→ **可靠**（输出敢信、不用逐条核）。＞省心放手 / 合我心意 / 扛大活（后三者排后，非不要）。

**N2 · 信任杀手**（可靠的反面，什么让你不敢信）：多选。
→ **四个全中**：① 谎报（说已改/已删/已测其实没做，B类）② 编造（假 API/文件名/事实，A类）③ 悄悄跑偏（擅自换方案不吭声，漂移）④ 假验证（看着对跑起来错、自证测试）。→ 可靠设计必须同时防这四类。

**N3 · 可靠的代价（友善度）**：一律从严还是看轻重？
→ **看轻重**（大事/危险动作严查，小改动别烦）。呼应全局「任务深浅自适应」。设计含义：harness 需要一把"轻重分类器"。

**N4 · 什么算"重/危险"（严查触发器）**：多选。（用户重选一次，最终=四个全中）
→ **删除/覆盖 + 外发/上线 + 碰核心 + 声称完成**（四选全中）。两种性质：前三类是**动手前**危险（删/发/碰核心，多可确定性检测 rm/push/核心 path → 前置拦/确认）；**声称完成**是**交付时**危险（事后**对账**核实，与 B类谎报硬闸同机制）。处置方式天然分两路。

**N5 · 动手前危险的处置**（删/发/碰核心）：
→ **分级**。不可逆的（删/发）**必须停下确认**；碰核心**给用户看清影响面即可放行**（不必硬拦）。声称完成走交付时对账（单列）。很"看轻重"。

**N6 · 结论必附证据（治编造 A类 + 假验证）**：
→ **分级**。交付/危险处必附真实证据（读到的工具返回 / 测试输出），琐碎处免。

**N7 · 偏离透明（治悄悄跑偏）**：
→ **大偏离才报**。小调整自主，方向性/换方案才必须先说。

### ★ 贯穿性「默认价值观」（7 问归纳，用于推导其余维度）
**看轻重、分级——大事/危险/交付处从严（甚至硬拦），琐碎处放手别烦。** 用户 N3-N7 每题都落在这条上，高度一致。可作未问维度的默认推导基准。

### 维度①「可靠/交付真实性」小结（= 九维的第 3+4 维，已细过）
- 谎报 → 交付时**确定性对账**（文件/git/exit 真实结果 vs 声称）。
- 编造+假验证 → **结论必附证据**（分级）。
- 危险动作(删/发/碰核心) → **分级处置**（不可逆必确认 / 碰核心看影响面）。
- 跑偏 → **大偏离才报**。
- 机制取向：硬闸只认铁证（近零误伤、可下沉）；空转类只能软（代理信号）；少而精、复用 pre-commit。

## 剩余 8 维系统访谈（用户选「都细问」，一维一维白话过）

### 维度②「接活前 · 澄清&范围」
**Q8 · 先问还是先做**（大而模糊的新活）：
→ **尽量先问**（拿不准就问、宁多问别做歪）。**比纯"看轻重"更保守**——"别做歪"权重高于"别烦"。适用面=真·新的/大而模糊的活；琐碎/续接轮仍不问（接 clarify_reminder hook 的负向 gate + 全局"过度提问=噪声"）。

**Q9 · 岔路处理**（干活中冒出 顺带做X / 前置卡路Y）：
→ **分类处理**（挡路小问题顺手解；大的/不挡路的先记下问你）。= 现有 focus 模型（前置阻塞/附加/无关/正当深入），看轻重。**维度②完成。**

### 维度③「干活中 · 方法纪律」
**Q10 · 根因优先 vs 先跑通**：
→ **必须先找根因**（禁止先加兜底/模糊匹配掩盖真问题）。比纯"看轻重"更严，质量>速度，同"澄清"维。= 现有 debugging §3 红线。

**Q11 · 改动克制**（外科手术 vs 顺手理旁边）：
→ **看情况**。危险/核心处外科手术式（只碰非改不可）；边角处可顺手理但**要告诉用户**。看轻重。= 现有 debugging"禁焦虑连带修改"红线的分级版。**维度③完成。**

### 维度④「卡住时 · 失败处置」
**Q12 · 鬼打墙阈值**（多快逼停）：
→ **试 3 次再停（第 4 次硬停）**。**★ 解掉 E-3**：阈值统一到"3 次/第 4 次硬停"（CLAUDE §8 版正确），debugging §6 的"≥2 次"应改齐（[[ghost-wall-threshold-conflict]] 就此有解）。

**Q13 · 停手后动作**：
→ **先问我**（停下+列已试方案+问用户要线索；用户大概率在场，便宜）。＞自动派外援。= 现有 escalate「选 A 问用户」优先。**维度④完成**（+空转=代理信号软提醒）。

### 维度⑤「跨会话 · 状态&知识」
**Q14 · 攒经验 memory 自主度**：
→ **主动写 + 当场报**（判断值得就写、写了啥告诉用户、可回滚）。高自主+透明，同"大偏离才报"调性。= 现有自动记 + summary 模式。

**Q15 · 上下文预算 · 硬拦 or 提醒**：
→ **只提醒不拦**。**★ 反着硬闸偏好** → 揭示原则细化：**硬闸只给"真相/危险"（谎报/删/发）；关于用户自己工作节奏的事（上下文满）要软、由用户拍板**。**设计含义**：现 CLAUDE §10 CRITICAL"95% 立即拒绝新任务"应**软化为只提醒**。snapshot/resume 自动存档保留（同 memory 自主度）。**维度⑤完成。**

### 维度⑥「知识治理 · 规则/文档」
**Q16 · harness 自治力度**（防规则肿）：
→ **自动盯 + 硬拦**（超标 pre-commit 阻断，逼当场清）。**不与 Q15 矛盾**：规则肿/死链/漂移是**机器可判死的事实**（近零误伤）→ 硬闸安全。**★ 原则再细化**：硬闸给"机器可判的事实（真相/危险/产品健康）"，软的留给"用户自己的判断"。**设计含义**：现有 rule_size_check（当前只提醒非阻塞）应**升级为 pre-commit 硬拦**（同 version_check 模式）。**维度⑥完成。**

### 维度⑦「协作模式」
**Q17 · 多 agent 倾向**：
→ **大活才叫**（大/复杂/要彻底才派多 agent；日常单干）。看轻重，= 现有 lvl0 默认 + 自动升级模型。**维度⑦完成。**

### 维度⑧「卫生 & 可移植」
**Q18 · dogfood/可移植 硬度**：
→ **机器可判的硬拦**（dogfood 漂移 / 换机会崩的配置，机关硬拦）。合"机器可判→硬"原则。**★ 定 E-6 方向**：enforce 镜像——把 CLAUDE §2 承诺的"镜像漂移检查 hook"做成 pre-commit 硬闸，缺的补装或显式 CHANGELOG 注明豁免。**维度⑧完成。**

### 维度⑨「人机沟通」
**Q19 · 沟通风格硬化？**
→ **不用**（保持软、靠自觉，不加机关）。同 Q15 逻辑：关于"用户自己领域"（沟通/节奏）保持软、用户拍板。沿用全局 CLAUDE.md 沟通 prefs。**维度⑨完成 → 九维全部访谈完毕。**

## ★★ 设计总纲（19 问汇总，2026-07-01 访谈成果）

**北极星**：可靠（输出敢信、不用逐条核）。

**贯穿性一条原则（用户自证一致）**：**硬闸只给「机器能一眼判死的事实」（真相 / 危险动作 / 产品健康）；软的留给「用户自己的判断」（上下文预算 / 沟通 / 节奏）。总体"看轻重"——大事/危险/交付处严（甚至硬拦），琐碎处放手别烦。** 但"澄清"与"根因"两处比纯看轻重更严（质量>速度）。

**九维落地**：
1. **可靠/交付真实**：谎报→交付时确定性对账硬闸（铁证）；编造/假验证→结论必附证据（分级）；危险动作→分级（删/发必确认、碰核心看影响面）；跑偏→大偏离才报。
2. **澄清&范围**：大而模糊尽量先问（比看轻重更保守）；岔路分类处理。
3. **方法纪律**：必先找根因（禁兜底）；改动看情况（危险处手术式）。
4. **失败处置**：第4次硬停（阈值=3）；停手先问我；空转=代理信号软提醒。
5. **跨会话**：memory 主动写+当场报；上下文只提醒不拦（软化现 §10 CRITICAL）；snapshot 自动。
6. **知识治理**：规则肿/死链/漂移→自动盯+硬拦（rule_size_check 升级为 pre-commit 阻断）。
7. **协作**：大活才叫多 agent。
8. **卫生/可移植**：dogfood 漂移/换机崩→机器可判硬拦（建镜像漂移 hook）。
9. **沟通**：全软/自觉，不加机关。

**机制取向（少而精）**：硬闸尽量复用 `.githooks/pre-commit`（version_check 同款）、少新建；软信号复用现有 reminder hook（context_warning 等）。下沉产品层 + dogfood 自身。

**顺带解决的搁置项**：★ E-3（鬼打墙阈值统一=3/第4次硬停）；★ E-6（dogfood enforce 硬拦方向）。

**下一步**：结合外部 48 方法（[[harness-trim-2026-07-01-deferred]] 关联的调研）→ 出正式设计 / 实施方案。
