## 核心教训

**描述性闸拦不住 agent**：`停下等用户点头`、`未确认不得进入下一步` 这类措辞是"描述"不是"命令"——agent 会解读成"我口头说一句，然后继续"，一口气把多道闸连跑完。

**工具级回合终止契约才有效**：在闸的位置写：
```
⛔ 硬契约：本步必须以 AskUserQuestion 调用结束当前回合，用户未回答前
禁止研读 / spawn 任何 agent / 往下走任何一步。
```

## 写闸的四要素（缺一不可）

1. **点名工具**：写出 `AskUserQuestion`，不能只说"停下/等用户"
2. **回合终止**：写出"结束当前回合"，agent 必须以该工具调用作为本条消息的终点
3. **前置禁止**：写出"用户未回复前禁止 [具体的后续动作]"——越具体越好（研读/spawn/编码）
4. **在正文里**：不能只写在文末"重要规则"区；正文各步骤处都要有，规则区最多是索引

## 相邻闸的额外问题

相邻两道闸（如闸①问问题、闸②复述确认）存在天然合并诱因：agent 拿到闸①回答后，会在同一回合做复述+继续。

**解法**：在闸②处明写「闸①②虽相邻，也**绝不可合并**为一回合连跑」。

## 适用场景

任何"多阶段重型 skill"里需要用户在烧 token 之前拍板的节点（研读前 / spawn 前 / 编码前）。当前已在 debate/collab 六步流程的三道闸实装。

**Why:** 这是 debate/collab v0.35.0 L1 审计中发现的最大漏洞，"指令可执行性"检验中最容易漏的点。

**How to apply:** 写或审任何 skill 的"确认步骤"时，对照四要素逐一检查。见 [[project-v035-state]]。
