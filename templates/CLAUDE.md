## 2.5 工具选择：检索用 Glob/Grep，执行才用 shell

**红线**：找文件 / 查内容 → 用 `Glob` / `Grep` / `Read`，**禁止**反射性用 shell 的 `find` / `Get-ChildItem` / `Select-String`（受控只读工具零弹窗 + 跨平台 + 不夹带删改）。干事情（构建 / 测试 / git / 进程）才用 shell——各司其职。

- **Glob 三诀**：① `path` 给具体别全盘扫（会超时）；② 匹配文件不匹配目录，找文件夹 `foo` 写 `**/foo/**`；③ 默认跳过 `.` 开头隐藏目录，目标在里面就把 `path` 扎进去。
- **禁止自拼 shell 批处理大检索 / 大输出**（`for` + 管道 / 多命令 `;` 串），一律拆成受控 `Grep` / `Glob` 单命令。**Why**：会触发"工具传结果线"概率性抽风（同段重影 / 命中 0 假空 / 幻影文件名），把脏数据喂进判断。
- **红线·脏数据上不拍板**：返回出现同段重影 / 命中 0 与预期矛盾 / 不认识的文件名 / `__unparsedToolInput` 时，**禁止**直接在该返回上下结论或改盘，先用 `wc -c` / `git status` / 受控 `Grep` 单命令二次验真。**口诀：dry-run 报的 N 处 ≠ 已改 N 处。**

## 5. Memory 项目内托管（自动）

memory 纳入项目 git（`.claude/memory/`），系统路径 `~/.claude/projects/<hash>/memory/` 用 junction 透明转发。由 `SessionStart` hook `memory_junction_check.py` 每次自动维护，无需手动。

> 红线：**绝不硬删可能含数据的目录**——任何一步失败即中止提示人工（迁移用改名 `.bak` 而非删除）。

**检索原则（主索引优先）**：① 先查 MEMORY.md 主索引（自动加载，≤40 条）；② 无匹配 → `/find-memory <关键词>` 搜冷区；③ **禁止**跳过主索引直接 grep 遍历 memory/（token 远高于 `/find-memory`）。MEMORY.md 是确定性派生索引，**勿手改**（会被重建覆盖）；换机 / junction SOP → `rules/portability.md` §2。

## 8. 鬼打墙觉察 + 渐进升级（红线）

**默认**：单 agent（lvl 0）。**自动升级我必须自觉触发，不等用户提**：

| From → To | 触发 | 我必须做 |
|-----------|------|---------|
| lvl 0 → 1（加 verification）| 等价性验收 / 重写 / 移植 | 完成后派**独立** verification agent，**禁止**自己写测试 |
| lvl 0 → 2（加 research）| 任务跨 ≥ 2 模块且我不熟悉代码 | 先派 Research agent 并行调研再动手 |
| lvl 1/2 → 3（升 4 角色）| **同一 bug 前 3 次改动失败，第 4 次禁止动手** | **必须停下**，列已试方案 + 未验证新假说 → 提议 `/escalate` 或 `/debate` |

### 红线

- **禁止**鬼打墙时闷头继续试（必须停下提议升级）
- **禁止用"我有新思路"豁免第 4 次硬停**。计数只看"同一症状下连续失败的改动次数"，不看思路是否不同——感觉"这次不一样"恰恰是规则最该绑住我的时候（高信心错误才是规则存在的理由）。
- **禁止**等价性任务自己写测试（自证陷阱）；**禁止**不熟悉的多模块任务不调研直接动手

### 计数规则（不留主观豁免）

- **同一 bug**=用户报同一症状（同一面板/字段/行为）计数 +1；**失败**=改完用户回报"还不对 / 还是 X / 依然 Y"。
- **每次失败必须**：动第 N+1 次前（N ≤ 3）先取一项**量化数据**（日志 / 产物 mtime / grep 结果 / 截图反推数值），没有就直接 `/escalate`，不准凭推断动手。**重置**=用户说"修好了 / OK 了"或切到不同 bug。
- `/escalate` 是用户急停按钮：万一我没自觉，用户按 → 强制总结 + 派外援 + 推荐路径。

## 9. 主观体验报告主动问范式（红线）

**触发**：用户基于外部观察报主观体验 / 怪现象，缺数据缺稳定复现（UI"有点黏" / API"偶尔返回空" / 后端"偶发 500"）。

**我必须一次性问全这 4 件（不要挤牙膏）**：① 可观察的证据？（截图 / 完整 input+output / request-response / 日志片段）；② 前 N 步操作 / 触发条件？；③ 多久发生一次？（偶发率 1/10 vs 1/100 决定打法）；④ 能存档现场吗？（若有 `/snapshot`）。

**接到信息后必须**：写 timer / counter / log **量化**（不只猜测）；教用户复现协议；接受概率方法（偶发率从 1/10 降到 1/100 也是修，允许"改一版观察 1 周"收尾）。

**禁止**：❌ 单凭描述就猜测修；❌ 一次只问 1 个挤牙膏；❌ 假装"我懂你的感受"绕开量化。

## 9.5 较大需求主动澄清 — `[clarify]` 信号响应

**触发**：`UserPromptSubmit` hook（`clarify_reminder.py`）在候选轮输出 `[clarify]`。读到后**语义精判**这轮够不够大，再决定问不问。

- **已表达执行意图、但范围 / 方案模糊**的较大需求 → **先用 `AskUserQuestion` 出 2-4 个结构化选择题**（带候选 + 推荐）对齐再动手。**禁止**对大需求零思考直接实现。
- **评估 / 咨询 / 可行性类问**（"能不能 X / 该不该 Z / 评估一下"这类只要结论的）→ 给结论 + 风险即**收口**。**禁止**把"建议"包装成 `AskUserQuestion` 执行菜单替用户开单——要不要执行由用户下一条 prompt 定。
- **琐碎 / 续接 / 已说全细节** → 直接执行。**禁止**对琐碎 / 续接轮强行问背景（过度提问 = 噪声）。
- **`AskUserQuestion` 文案约束**：选项 / 入参必须短、纯文本、避免大段中文（长中文 → `\u` 长串是 `__unparsedToolInput` 转义炸的诱因）。

> 混合判定分工 / hook 路径 / 注册 / 调参 → `rules/anti_drift_hooks.md` §1。

## 9.6 任务防漂移 — `[focus]` 信号 + 漂移分类响应

**触发**：`UserPromptSubmit` hook（`focus_reminder.py`）把本会话首条实质 prompt 记为「任务锚」，周期注入 `[focus]`。可用 `/focus` 查看 / 改 / 清。读到后判断这轮还属不属于原任务，**若偏离按类型分发，禁止闷头做**：

| 漂移类型 | 判据 | 响应 |
|---------|------|------|
| **前置阻塞** | B 必须先解才能回原任务 | **大** → `/spinoff` 交接 + 提示开新对话；**小**（几行）→ 内联做完即回 |
| **附加扩张** | "顺带也做 X"，不阻塞 | `/todo` 归档，先把原任务做完 |
| **无关支线** | 跟原任务无关的新话题 | `/todo` 或建议开新对话，**绝不内联** |
| **正当深入** | 仍是原任务合理子步骤 | 忽略 `[focus]`，继续做 |

### 红线

- **禁止**发现"得先做别的"就地闷头展开——先判类型再响应；**禁止**对小前置 / 不阻塞的附加滥用 `/spinoff`（`/spinoff` 只给"够大的阻塞性前置"）。
- **禁止**反漂移机制自己变成漂移源——`[focus]` 是周期提醒不是每轮命令，命中"正当深入"就忽略；真漂没漂最终由我语义判断。

> 主动+被动入口 / 锚文件 / hook 路径 / 调参 / 配套 skill → `rules/anti_drift_hooks.md` §2。

## 10. 上下文预算红线 — `[ctx-budget]` 信号响应

**触发**：`UserPromptSubmit` hook（`context_warning.py`）每轮估算用量，超阈值输出 `[ctx-budget]`（char/4 启发式，±10%）。我**必须**按级响应：

| 级别 | 用量 | 行为 |
|------|-----|------|
| 无 | < 75% | 正常执行 |
| **MEDIUM** | 75-84% | 执行，**完成后**建议 "/snapshot 锁状态准备换会话" |
| **HIGH** | 85-94% | **响应开头**告知用量 + 建议 /snapshot + 询问是否继续；**只接受小任务**，拒绝复杂多文件改动 |
| **CRITICAL** | ≥ 95% | **立即拒绝**新任务，告知"上下文几乎满，先 /snapshot 再开新会话 /resume 接续" |

### 红线

- **CRITICAL 下禁止开始任何新任务**——即使用户说"快做"也拒绝（只允许 `/snapshot` 这类保命操作）；**HIGH 下禁止启动复杂多文件改动**。
- 边界附近以信号为准，不要二次判断"我感觉还行"；**不要悄悄忽略信号继续做事**。
- `/snapshot` / `/resume` / `/git-sync` 等 slash 已被 hook 自动豁免（否则保命操作被拦会死锁）。

> hook 路径 / 注册 / 调参（`WINDOW` / `THR_*`）/ 豁免论述 → `rules/anti_drift_hooks.md` §3。

## 11. 文档管理（红线）

**本项目必须用 `doc/` 分层管理所有文档，结构按 `rules/workflow.md §6` 六层定义**（序号前缀是排序约定）：

```
0_architecture（架构红线+acceptance+TODO） / 1_plan（活跃推进） / 2_pending（未决备忘）
3_design（实现设计） / 4_archive（归档） / 9_reference（外部参考）
```

### 红线

- **禁止**文档散落项目根目录或乱挂（设计文档放 `doc/3_design/` 而非 `src/<module>/README.md`）
- **禁止**跳过六层任何一层（即使空，目录也要在 — `.gitkeep` 占位）；**禁止**给六层改名 / 合并；**禁止**新增同级目录（如 `doc/notes/`）——新需求按 §6 边界归到现有六层
- **强制** `doc/README.md` 作唯一索引，任何 `doc/*.md` 的新增 / 删除 / 重命名都要同步它

**不接受 doc/ 强制 → 不应使用 bridgeforge**（改用更宽松的脚手架）。

> **Why + 每层职责边界 + README 同步细则** → `rules/workflow.md §5.5-§8`。
