## 3. 禁止事项

| 层 | 职责 | 禁止 |
|----|------|------|
| 协调中枢（如 `core/` / `app/`） | 协调 + 基类 + 横切服务 | 放具体功能逻辑 |
| Feature 模块 | 独立功能 | import 其他 feature 的内部实现 |

**判断标准**：被 2 个以上 feature 模块使用的代码 → 抽到协调中枢的 `utils/` 或 `base/`；只被单个模块使用的 → 留在该模块目录下。

### 3.1 协调中枢内部分层（红线）

协调中枢（`core/` / `app/core/` / 同等位置）内部必须按职责分三块，**禁止**直接堆杂项文件，也**禁止**在仓库根放横切服务：

| 子目录 | 放什么 | 禁止 |
|--------|--------|------|
| `<core>/config.py` 或 `<core>/constants.py` | 跨 feature 共用的纯常量（路径 / URL / 配置 key） | 任何副作用（`load_dotenv` / 读 env / SDK import / IO）|
| `<core>/bootstrap.py` | 幂等的运行时引导函数（`ensure_env()` 等） | 把状态写成模块级副作用（必须包成函数 + flag 幂等） |
| `<core>/services/` | 横切服务模块（被多个 feature 调用的服务：logger / tracker / 自更新 / 健康检查等） | 任何只被单 feature 用的工具（应留 feature 目录） |

**Why**：纯常量文件混入副作用 import → 非该 feature 的调用点也会被拖累启动时间 + 拉起不需要的依赖。集中到中枢分层让"共享 vs 私有"边界一眼可见，副作用全部在 `bootstrap` 内显式调用而不是 import 时静默触发。

### 3.2 提炼共享常量的迁移范式

把某 feature 模块里的共享常量提炼到中枢时，**禁止**简单搬常量；必须三件套：

1. **纯常量文件**：常量搬到中枢的 `config.py` / `constants.py`（零副作用）
2. **幂等引导函数**：原模块级副作用包成中枢的 `bootstrap.py::ensure_env()`（重复调零开销，靠 flag 防重入）
3. **原模块改 re-export + 调引导**：保持向后兼容，旧 `from old.module import X` 调用点不破

迁移过程中**任一步漏做**都会留下隐患：只搬常量 → 副作用消失导致下游运行时报错；不做 re-export → 大量调用点要同步改。

## 4. 运行时数据目录（红线）

**所有运行时数据 / 缓存 / 日志 / 下载 / 安装的二进制必须放 `.runtime/` 下，不在项目根目录散落。**

| 子目录（示例） | 内容 | 是否 git track |
|---------------|------|---------------|
| `.runtime/<tool>.exe` | 嵌入式工具（同事零配置部署用） | ✅ track |
| `.runtime/<cache-name>/` | 预下载缓存（脱网恢复需要） | ✅ track |
| `.runtime/output/` | 程序写出的报告 / 截图 / 中间产物 | ❌ gitignored |
| `.runtime/logs/` | 启动脚本 / 应用运行日志 | ❌ gitignored |

**Why**：避免根目录塞一堆 `output/` `logs/` `cache/` 视觉噪声，统一 `.runtime/` 让"代码 vs 运行时副产物"一眼分清。

**How to apply**：

- 新增"输出目录"类常量时（如 `OUTPUT_DIR` `CACHE_DIR`），必须挂 `.runtime/` 下，禁止直接挂项目根
- `.gitignore` 用 `.runtime/<subdir>/` 精确忽略子目录，**不要**整个 `.runtime/` 一刀切（会误伤需 track 的嵌入式工具 / 预下载缓存）；入口脚本日志也走 `.runtime/logs/`

## 5. 根目录极简（红线）

**项目根目录如无必要不要新增文件或文件夹。** 新增任何根级条目前，必须先问"它能不能挂到现有顶层目录下？"

**判定标准**：

- 一次性脚本 / 调研脚本 / demo / 探针 → 全部进 `scripts/`，**禁止**另起 `tools/` `probes/` `demos/` 等同义目录
- 运行时副产物（缓存 / 日志 / 输出）→ 全部进 `.runtime/<subdir>/`（见 §4）
- 文档 → `doc/<分类>/`
- 配置 → `.claude/` 或 `.<tool>/`（dotfile 配置除外，如 `.gitignore` `.gitattributes` 这类工具规定根级的可保留）
- 本地包源码 → `libs/<pkg>/`

**禁止**：

- 因为"这堆文件跟现有目录不太搭"就在根新建 `misc/` `extras/` `temp/` 这类兜底目录 — 这是范畴漂移的开始
- 同义目录共存（`scripts/` 和 `tools/` / `output/` 和 `.runtime/output/`）

**Why**：根目录是项目的门面，每多一个顶层条目都在稀释"这是个什么项目"的第一印象。同义目录还会让"放哪个"成为反复的认知摩擦。

**How to apply**：

- AI 自己造新目录前必须先 `ls` 根目录，确认无可挂靠的现有目录
- 用户要求新增工具型脚本/demo/探针时，默认放 `scripts/`，不要主动建新顶层目录
- 发现根目录出现同义目录时，主动提议合并
