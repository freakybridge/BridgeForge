# 调试规范

> 适用场景：排查 bug、分析日志、编写调试脚本时

---

## 1. 核心检查项

<!-- TODO: 按本项目实际的并发模型 / 数据流补充 -->

| 领域 | 检查项 |
|------|--------|
| **异步/并发** | 同步调用阻塞事件循环、未 await 的任务、CPU 密集任务用线程池 |
| **状态一致性** | 本地缓存与数据源同步、浮点数精度 |
| **数据流** | 各层之间的事件路径、信号槽连接、高频数据的 UI 节流 |
| **并发竞态** | UI 线程与计算/网络线程的竞态 |

---

## 2. 约束

- **提交前检查**：`git add` 前必须确认无临时文件混入（`git status` 核查）
- **日志自读**：调试后必须主动读取日志，直接给出结论，**不得**让用户手动粘贴日志
- 调试涉及外部副作用（订单 / 邮件 / 支付）时，必须确认 Mock / Sandbox 模式已开启

---

## 3. 修 bug 前必须确认根因

**禁止**在未确认根因的情况下添加 fallback、兜底逻辑、模糊匹配等"保险"代码。这类补丁掩盖真实问题，后续更难排查。

1. **先追踪完整调用链**：从触发点到出错点，逐层确认数据在哪一步变错
2. **确认根因后再动手**：如果不确定，向用户提问（"这个数据是从哪来的？""这个操作之前做了什么？"），不要猜
3. **禁止先加后删**：不要"先加个兜底跑通，回头再查根因"——回头往往不会发生
4. **根因未确认必须显式标置信度**：动手前若根因仍是假说，须一句话标出**置信度 %**（低于 50% 直说"不确定"）；**禁止**把低置信假说包装成"就是它了"的结论下手——低置信就先验证或问用户，别赌。

## 4. 跨语言 / 跨层级崩溃诊断顺序（红线）

**access violation / 进程闪退** 时，高级语言（Python / Node 等）的 traceback 常是误导。它给出的是"崩溃瞬间各线程碰巧在哪里"的快照，不是因果。

**禁止**基于高级语言栈帧推断"是 XX 线程 race / 信号桥接问题"就改高级语言代码。必须走以下诊断顺序：

1. **先在 FFI 边界加日志**。每个跨语言调用前后各一条。"前"有"后"无 → 锁定在 native 内部
2. **核实 native binary 真实版本**（用 SDK 自带的 GetVersion API，不要信文件名或包版本号）
3. **用 SEH `__try/__except`（Win）或 signal handler（Unix）包 native 调用**，把崩溃收敛成日志 + 错误码
4. **排除 FFI 层之后**，才去怀疑高级语言侧的线程 / 信号 / 框架

## 5. 修复前必须确认影响范围

写修复代码前，先列出这个改动的影响范围并确认覆盖：

- **哪些数据源受影响？**
- **哪些用户路径走这条逻辑？**
- **边界情况**：空值、零值、无数据、跨地域 / 跨币种 / 跨时区
- 如果无法确认某个边界情况，**主动问用户**，不要默认"应该没事"
- **交付前改动面自证**：若单次改动 **> 8 个文件或 > 400 行**，响应里必须贴 `git diff --stat` 并**逐文件说明为何非改不可**——大重构正当超标也要说清，不默认"顺理成章"。
- **顺手改必告知（Q11 红线）**：任何**非用户本次点名**要改的顺手改动——哪怕只碰 1-2 个文件、够不上上面的量阈值——都必须在响应里主动一句话「顺带改了 X，因为 Y」，**绝不静默夹带**（与 `CLAUDE.md §9.6` 顺手改红线呼应）。

---

## 6. 鬼打墙检测与升级（红线）

当陷入"反复无效尝试"状态时，**必须立即停手**，转向求助或辩论。反复发出未验证的修复给用户是**严重违规行为**——它把调试成本甩给用户并污染 baseline。

### 触发条件（任一命中即必须升级，不需"感觉"）

| # | 条件 | 说明 |
|---|------|------|
| **T1** | 同一 bug 连续改代码 **≥ 3 次**仍未解决（**第 4 次禁动手**） | 计数包含"改完但没验证"的尝试，回滚不算。**计数唯一权威 = CLAUDE §8**（阈值以 §8 为准，勿在此另立数） |
| **T2** | 用户说过"还崩 / 还错 / 依然有问题" **≥ 2 次** | 每次都是"你发出的修复没用"的信号 |
| **T3** | 当前诊断基于**栈帧/症状描述**而非**量化数据**（版本号 / md5 / 日志时序 / dumpbin 输出） | native crash 尤其要警惕 |
| **T4** | 出现**对称反证**（"环境 A 能 / 环境 B 不能"、"昨天能 / 今天不能"）但**当前假说无法解释**这个不对称 | 对称反证是金矿，无法解释 = 假说错了 |
| **T5** | 正在改的文件和 crash 发生的**运行时层级相差 ≥ 1 层**（例：症状在高级语言但我在改高级语言；真因可能在 native FFI） | 跨层改动必须先用诊断日志确认因果 |
| **T6** | 累计改过**≥ 3 处代码**且没有回滚前面的 | 每次新改都在增加噪音污染。**计数以 CLAUDE §8 为准** |
| **T7** | 用户直接问"你自己调试过吗"/"你验证过吗" | **最终警报**，立即停手且大声承认 |

### 触发后必须立即执行

**Step 1 — 停手**：
- 不再写代码、不再发修复
- 如果前几轮改动已经 ship 但没效果，立即 `git diff` 列出所有未 commit 的改动

**Step 2 — 在"问用户"和"启动 debate"之间二选一**：

| 选 A（主动问用户） | 选 B（启动 `/debate`） |
|---|---|
| 我可能缺用户才知道的信息（配置、环境、历史操作、日志片段） | 我有足够信息但假说互相冲突 |
| 还没有复现环境 | 已经能复现但不知道怎么修 |
| 用户能在 1 分钟内回答的问题 | 需要系统性重新审视根因 |

**当不确定选 A 还是 B 时，优先选 A**。问用户比自己空转便宜 10 倍。

**Step 3 — 问用户的格式（选 A 时）**：
1. **当前症状**一句话
2. **我已经尝试过什么**（列已改动的文件 + 结果）
3. **我对根因的当前假说**（附置信度 %，低于 50% 必须承认）
4. **我需要的信息**（问具体问题，不要问"你觉得呢"）
5. **如果你不知道，我建议走 debate** —— 给用户一个 fallback 选项

**Step 4 — 启动 debate 的前提（选 B 时）**：
- 先写好 `doc/2_pending/debates_YYYY-MM-DD_<topic>.md` 的 problem statement，列出已确认事实和分歧点
- 调用 `/debate` skill（如果项目装了），带上这个文档的路径和核心疑问
- **Round 1 的 reviewer 必须真正动手跑验证命令**（不能只做纸面推导）
- 辩论达成共识**前**禁止写任何生产代码

### 禁止事项

- **禁止"先加个兜底跑通，回头再查根因"**
- **禁止发给用户"应该能修"的改动**。每一版修复在发送之前必须自己能复现原 bug 并验证修复后不再出现
- **禁止跨 ≥ 2 层同时改动**。每次只动一层，改动前先加对应层的诊断日志
- **禁止忽略对称反证**。出现"A 能 / B 不能"时，必须在动代码前写出"我的假说如何解释这个不对称"，写不出来就是假说错了
- **禁止基于"焦虑心态"的连带修改**。修 A bug 时顺手改一堆"看起来可能相关"的 B/C/D 地方——这种是焦虑驱动的，不是证据驱动的，全部算违规

## 7. "改完代码 → 让用户测试" 前必做的验证（红线）

**触发**：改动了**会编入运行时产物**的代码后（native binary / 容器镜像 / build 缓存），让用户运行验证**之前**。

### 必做的 3 步验证

1. **构建检查**：`<build-command>` 输出 success，且无 `warning: file modified` 类提示
2. **mtime 检查**：构建产物的 mtime 必须**晚于**任何被修改的源文件
3. **字面量嵌入检查**：每次代码改动加唯一标识字符串到日志（如 `[fix-{{TODAY}}-X]`），改完用 `grep -ac "<标识>" <build-product>` 必须 **≥ 1**

三步全过 → 才让用户启动验证。任一步失败 → **暂停让用户操作 → 排查工具链**。

### 禁止

- ❌ 改完代码直接让用户"重启看看"——这是反复撞墙的根因之一
- ❌ 用源码 mtime 推断产物含新代码（mtime 只证明源码改了，不证明 link 进产物）
- ❌ "用户应该会自动重新构建"——某些 build 工具在 cache 异常时会静默 fallback 旧产物

## 8. 复现不稳定的 bug：先重启再分析

当静态分析代码逻辑无误但 bug 仍出现时，**优先重启进程验证**，再决定是否继续调试。

原因：长运行进程会积累状态（脏缓存、死定时器、内存碎片），重启可清除大量隐性问题。

**触发条件**：同一 bug 分析 2 轮以上未找到明确根因，且症状不稳定（有时正常有时不正常）。

---

## 9. 文件归属规范

| 文件类型 | 存放位置 | 禁止 |
|---------|---------|------|
| 配置文件 | `config/` | 放根目录 |
| 调试/临时脚本 | `tests/output/` 或 `scripts/dev/` | 放根目录或业务目录 |
| 备份/日志/tmp | `tests/output/` | 根目录生成 `.bak`/`.tmp`/`.log` |
| 文档 | `doc/` 对应子目录 | 放根目录 |

---

## 10. 性能调优必须先量化

性能优化**禁止**连续猜测式修改。第一步必须加诊断/profiling 量化瓶颈。

1. 加计时/计数工具，量化各环节耗时
2. 数据指向哪里就优化哪里，不凭直觉
3. 优化后用同样的工具验证效果
4. 诊断工具用完后**必须清理**（它本身也有性能开销）

---

## 11. 响应结构（调试场景）

1. **预测**：最多 3 条根因分析
2. **调查**：最多 3 条验证步骤
3. **解释**：根因 + 行号
4. **修复**：仅输出修改的类/函数，标记 `[MODIFY]`/`[NEW]`/`[DELETE]`
5. **验证**：提供测试命令或日志观察命令
