## 4. 重写 / 移植期间的功能等价性（红线）

> 仅当本项目存在"老版本 → 新版本"重写时启用本节，否则可删。

**新版本必须完整实现旧版本的对应功能。**

### 禁止"先绕过后补"

- **禁止**用 `// TODO` / `// For now` / 临时方案跳过已有功能
- **禁止**在上层内联实现本应由下游模块承担的逻辑
- 如果某功能在新架构中确实无法实现或需要不同设计，**必须立即向用户报告**，不得静默跳过

### 重写验收检查清单

每个模块重写完成后，必须对照旧版本逐项确认：

| 检查项 | 说明 |
|--------|------|
| **功能完整** | 旧版本中该模块的所有 public 方法在新版中都有对应实现 |
| **链路完整** | 调用链路与旧版一致，不存在绕过中间层的捷径 |
| **事件回调** | 所有相关事件类型都触发了该模块的事件处理 |
| **接口暴露** | 上层调用接口与旧版本兼容，签名一致 |
| **参数透传** | 所有字段都被正确解析和使用，不得硬编码默认值 |

## 5. 模块组织约束

完整范式见 `rules/modules.md`。核心原则：

- **新增功能** 必须创建为独立 feature 目录，**禁止**在协调中枢类目录（如 `core/` / `app/`）下新增功能文件
- 协调中枢只放：协调器 + 基类 + 横切服务
- 模块间**禁止**互相 import 对方内部实现
